# Airflow Lite - 요구사항 정의서

## 1. 프로젝트 개요

### 1.1 배경

생산 시스템 환경에서 Oracle 11g의 생산 데이터를 Parquet 파일로 이관하는 파이프라인이 필요하다. Apache Airflow는 이 용도의 사실상 표준이지만, Windows Server + Oracle 11g 단일 서버 환경에서는 설치/운영 복잡도가 과도하다.

### 1.2 목표

Apache Airflow의 **핵심 개념 3가지**(파이프라인 단계 정의, 재시도/에러 핸들링, 백필)만 차용하고, 나머지는 경량 라이브러리로 대체하여 생산 시스템 환경에 맞는 파이프라인 엔진을 구축한다.

### 1.3 핵심 가치

- **선언적 파이프라인 정의**: 추출 → 변환 → 적재 → 검증 단계를 명시적으로 정의하고 의존성 제어
- **자동 장애 복구**: Oracle 연결 끊김 등 생산 시스템 환경 특유의 오류에 대한 자동 재시도
- **안전한 재처리**: 과거 데이터를 멱등성 보장하며 재이관

---

## 2. 운영 환경

| 항목 | 사양 |
|------|------|
| OS | Windows Server 2019 |
| 소스 DB | Oracle 11g (사내 서버) |
| 실행 형태 | 단일 서버, 단일 프로세스 |
| 네트워크 | 사내망 (인터넷 미연결 가능) |
| Python | 3.11+ |

### 2.1 환경 제약사항

- Linux 전용 도구(Celery, Redis 등) 사용 불가
- Oracle 11g용 드라이버(oracledb) 호환 필수
- Windows 서비스로 장기 운영 가능해야 함
- 클라우드 서비스 의존 없음

---

## 3. 기능 요구사항

9개 기능 영역을 Airflow 대비 3가지 전략으로 분류한다.

### 3.1 핵심 차용 (Airflow 개념 벤치마크) — 3개 영역

#### FR-01. 파이프라인 단계 정의

- 파이프라인을 복수의 단계(step)로 분리하여 선언적으로 정의할 수 있어야 한다
- 각 단계 간 의존성(순서)을 명시할 수 있어야 한다
- 특정 단계의 성공/실패에 따라 후속 단계의 실행 여부를 제어해야 한다
- 단계 단위로 실행 상태(대기/실행중/성공/실패/스킵)를 추적해야 한다

#### FR-02. 재시도 및 에러 핸들링

- 태스크 실패 시 설정 가능한 횟수만큼 자동 재시도해야 한다
- 재시도 간격(고정/지수 백오프)을 설정할 수 있어야 한다
- Oracle 연결 끊김, 네트워크 타임아웃 등 MES 환경 특화 에러를 식별하고 적절히 처리해야 한다
- 최종 실패 시 콜백(알림 발송 등)을 실행할 수 있어야 한다

#### FR-03. 백필 (재처리)

- 과거 특정 날짜/기간을 지정하여 파이프라인을 재실행할 수 있어야 한다
- 재처리 시 기존 데이터에 대한 멱등성을 보장해야 한다
- 월별 Parquet 파티션 단위로 안전하게 덮어쓰기해야 한다
- 재이관 전 기존 파일 백업을 수행해야 한다

### 3.2 경량 대체 — 4개 영역

#### FR-04. 스케줄링

- Cron 표현식 기반의 주기적 실행을 지원해야 한다
- 고정 간격(Interval) 기반 실행을 지원해야 한다
- Windows 서비스로 스케줄러를 안정적으로 운영할 수 있어야 한다
- 수동 즉시 실행(trigger)을 지원해야 한다

#### FR-05. 모니터링 UI

- 파이프라인 실행 이력을 조회할 수 있어야 한다
- 각 단계별 상태(성공/실패/진행중)를 시각적으로 확인할 수 있어야 한다
- 실행 시간, 처리 건수, 에러 메시지 등 상세 정보를 제공해야 한다
- 기존 React 프론트엔드에 대시보드로 통합해야 한다

#### FR-06. 로깅 및 감사 추적

- 파이프라인 단계별 로그를 기록해야 한다
- 실행 결과(시작/종료 시간, 처리 건수, 에러 내용)를 영구 저장해야 한다
- 로그 파일을 날짜별로 로테이션해야 한다
- 실행 이력을 DB에 구조화하여 저장해야 한다

#### FR-07. 알림 (Alerting)

- 파이프라인 실패 시 이메일로 알림을 발송할 수 있어야 한다
- 사내 메신저 웹훅을 통한 알림을 지원해야 한다
- 알림 대상과 조건을 설정할 수 있어야 한다

### 3.3 제외 — 2개 영역

#### FR-08. 분산 실행 — 제외

- 데이터 이관은 단일 서버 순차 처리로 충분하다
- Oracle → Parquet 변환은 I/O 바운드이므로 분산보다 배치 크기 최적화가 효과적이다

#### FR-09. 플러그인 생태계 — 제외

- 연동 대상이 Oracle 11g 단일 DB이므로 범용 커넥터 생태계가 불필요하다
- Oracle 드라이버(oracledb)를 직접 사용한다

## 4. 비기능 요구사항

### 4.1 성능

- NFR-01: 단일 테이블 월별 데이터(100만 건 기준) 이관을 30분 이내에 완료해야 한다
- NFR-02: 파이프라인 엔진 자체의 메모리 사용량은 500MB 이내여야 한다
- NFR-03: 대용량 데이터 처리 시 청크 단위 스트리밍을 지원하여 메모리 폭주를 방지해야 한다

### 4.2 안정성

- NFR-04: 스케줄러가 비정상 종료 후 재시작 시, 미완료 작업의 상태를 복구할 수 있어야 한다
- NFR-05: Oracle 연결 장애 시 자동 재시도 후 복구되어야 한다
- NFR-06: Windows 서비스로 등록하여 서버 재부팅 시 자동 시작되어야 한다

### 4.3 유지보수성

- NFR-07: 새로운 파이프라인(테이블) 추가 시 설정 파일 또는 코드 최소 변경으로 가능해야 한다
- NFR-08: 파이프라인 단계를 독립적으로 테스트할 수 있어야 한다
- NFR-09: 핵심 모듈의 단위 테스트 커버리지 80% 이상을 유지해야 한다

### 4.4 보안

- NFR-10: Oracle 접속 정보(비밀번호 등)를 코드에 하드코딩하지 않아야 한다
- NFR-11: API 엔드포인트는 사내망 내부에서만 접근 가능해야 한다

## 5. 기술 스택

| 영역 | 기술 | 용도 |
|------|------|------|
| 언어 | Python 3.11+ | 전체 |
| 파이프라인 엔진 | 자체 구현 | DAG 개념의 단계 정의 및 상태 관리 |
| 스케줄링 | APScheduler | Cron/Interval 기반 주기 실행 |
| 재시도 | Tenacity | 자동 재시도, 지수 백오프 |
| API 서버 | FastAPI | 수동 트리거, 백필 요청, 상태 조회 |
| 실행 이력 DB | SQLite | 경량 메타데이터 및 실행 이력 저장 |
| Oracle 연동 | oracledb | 소스 DB 접속 |
| 데이터 포맷 | Parquet (Snappy 압축) | 이관 결과 저장 |
| 프론트엔드 | React + AG Grid + Recharts | 모니터링 대시보드 (2단계) |

- ** 기술 스택에 명시된 모든 기술은 운영 환경에서 지원하는지 꼭 점건해야 한다. 예) cx_Oracle은 Python 3.14에서 설치 불가 (개발 공식 종료)**

## 6. 단계별 구현 범위

### 6.1 1단계 — 파이프라인 엔진 코어

> 목표: 코드로 파이프라인을 정의하고 실행할 수 있는 최소 엔진

- FR-01: 파이프라인 단계 정의 및 상태 관리
- FR-02: 재시도 및 에러 핸들링
- FR-03: 백필 (재처리)
- FR-06: 기본 로깅 (파일 + SQLite 이력)
- NFR-07~09: 테스트 가능한 구조

### 6.2 2단계 — 스케줄러 + API + 알림

> 목표: 자동 실행과 외부 인터페이스

- FR-04: APScheduler 기반 스케줄링
- FR-07: 이메일/웹훅 알림
- FastAPI 기반 API (수동 트리거, 백필 요청, 상태 조회)
- Windows 서비스 등록

### 6.3 3단계 — 모니터링 UI

> 목표: 운영 가시성 확보

- FR-05: React 대시보드 통합
- 실행 이력 그리드, 상태 시각화
- 실시간 로그 조회

---

## 7. 제외 범위

다음 항목은 본 프로젝트 범위에 포함하지 않는다.

| 항목 | 사유 |
|------|------|
| 분산 실행 (Celery, Kubernetes) | 단일 서버 환경, I/O 바운드 워크로드 |
| 플러그인/커넥터 생태계 | Oracle 11g 단일 소스 |
| DAG 시각적 편집기 | 코드 기반 정의로 충분 |
| 클라우드 연동 (S3, GCS 등) | 사내망 폐쇄 환경 |
| 실시간 스트리밍 처리 | 배치 처리만 대상 |
