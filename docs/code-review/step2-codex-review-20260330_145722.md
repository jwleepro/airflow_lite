# Step 2 Codex 재검토

> 작성일: 2026-03-30 14:57:22
> 대상: `docs/code-review/step2-review.md`
> 목적: Claude Code 리뷰 내용의 사실관계 재검증

## 결론

리뷰 문서의 핵심 우려 중 일부는 타당하지만, 몇 항목은 현재 구현과 맞지 않거나 다른 모듈의 동작과 혼동되어 있다.

## 사실과 다른 항목

### [P0-1] `ParquetWriter.restore_backups()` 파일명 변환 버그

- 현재 `ParquetWriter.backup_existing()` 는 `.parquet -> .bak` 로 변경한다.
- 따라서 `restore_backups()` 의 `.bak -> .parquet` 복원은 현재 구현과 일관된다.
- 리뷰 문서의 `.parquet.bak` 예시는 `ParquetWriter` 가 아니라 `BackfillManager` 구현과 혼동한 것으로 보인다.

근거:
- `src/airflow_lite/transform/parquet_writer.py`
  - `backup_existing()`: `parquet_file.with_suffix(".bak")`
  - `restore_backups()`: `bak_file.with_suffix(".parquet")`

### [P1-3] `FullMigrationStrategy._had_backup` 타입 혼동

- 현재 `_had_backup` 에 저장되는 값은 `ParquetWriter.backup_existing()` 의 반환값이며 타입은 `bool` 이다.
- `Path | None` 반환은 `engine/backfill.py` 의 `BackfillManager.backup_existing()` 쪽 구현이다.
- 따라서 이 항목은 현재 `FullMigrationStrategy` 기준으로는 사실이 아니다.

## 표현 수정이 필요한 항목

### [P1-2] `finalize_run` 예외 무시

- 후처리 실패 예외가 상태로 승격되지는 않지만, 완전히 조용히 지나가지는 않는다.
- 현재 `PipelineRunner.run()` 은 `logger.exception(...)` 으로 예외를 기록한다.
- 따라서 "조용히 지나감" 보다는 "로그만 남고 상태/알림에 반영되지 않음" 이 더 정확하다.

### [P1-5] unique 인덱스 제거 후 중복 성공 레코드 방지 로직 확인 필요

- 현재 저장소 조회 로직은 이미 최신 성공 1건만 반환하도록 구현되어 있다.
- `find_latest_success_by_execution_date()` 는 `ORDER BY created_at DESC LIMIT 1` 을 사용한다.
- 이 항목은 잠재 리스크 메모 정도는 가능하지만, 현재 코드상 미확인 상태는 아니다.

### [P2-1] `IncrementalMigrationStrategy` 롤백 불완전

- 현재 증분 적재는 기존 파일 직접 append 가 아니라 sidecar 파일 추가 방식이다.
- 실패 시 `_written_files` 에 기록된 신규 파일만 삭제하므로, 리뷰 문서의 "기존 파일에 이미 append된 데이터는 복구 불가" 시나리오는 현재 코드 경로에 직접 해당하지 않는다.
- 다만 verify 자체가 약하다는 별도 문제는 존재한다.

## 타당한 지적

### [P0-2] `IncrementalMigrationStrategy.verify()` 검증 약함

- 현재 로직은 `self._loaded_count <= parquet_count` 이다.
- 이 비교는 전체 파티션 row 수가 커질수록 검증 의미가 약해진다.
- 리뷰 문서의 문제 제기는 타당하다.

### [P1-1] `force: bool = True` 기본값

- 현재 manual trigger 와 backfill 모두 기본값이 `True` 다.
- 이것이 버그인지 여부는 정책 문제지만, 수동 트리거의 안전 기본값 관점에서는 재검토할 만하다.
- 다만 이 값은 현재 API 테스트가 명시적으로 기대하고 있으므로 변경 시 계약과 테스트를 함께 수정해야 한다.

### [P2-3] `retry_count` 최종 시도 횟수 부정확

- 현재 `retry_count` 갱신은 `before_sleep` 훅에서만 수행된다.
- 최종 실패 시 마지막 attempt 수가 DB에 정확히 반영되지 않을 수 있다는 지적은 타당하다.

### [P2-4] DB 마이그레이션 트랜잭션 미흡

- `sqlite3.executescript()` 는 중간 실패 시 앞선 문장이 남을 수 있다.
- 따라서 부분 완료 상태 가능성 지적은 타당하다.

## 테스트 관련 정정

- `restore_backups` 동작은 이미 테스트가 존재한다.
  - `tests/test_extract.py::test_restore_backups_removes_new_outputs_and_restores_originals`
- verify 실패 시 run 상태 실패 전환과 failure callback 경로도 일부 테스트가 있다.
  - `tests/test_service.py::test_factory_marks_run_failed_when_verify_returns_false`
  - `tests/test_service.py::test_factory_on_failure_callback_calls_alert_manager`
- `force=True` 기본 동작도 현재 API 테스트가 기대값으로 고정하고 있다.
  - `tests/test_api.py::test_trigger_pipeline_success`
  - `tests/test_api.py::test_backfill_success`

## 추가 메모

- 이번 재검토 중 `Path.with_suffix()` 동작과 SQLite `executescript()` 동작은 별도 로컬 재현으로 확인했다.
- pytest 전체 재실행은 시도했지만, 이 환경에서는 임시 디렉터리 정리 단계에서 Windows 권한 오류가 발생해 최종 통과 여부는 재검증하지 못했다.
