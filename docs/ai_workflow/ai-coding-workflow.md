# AI Coding Workflow: GPT-5.3 개발 방식 → 프레임워크 개발 매핑

> GPT-5.3의 AI-assisted model development 파이프라인을 일반 프레임워크 개발(airflow-lite)에 매핑한 분석

---

## 8단계 매핑

### 1단계: 사람이 큰 틀을 잡는다

**GPT-5.3에서**: 연구팀이 "코딩 성능 향상", "에이전트 능력 강화" 같은 목표를 설정

**airflow-lite에 대응하면**:
- "Oracle→Parquet 마이그레이션을 500MB 메모리 안에서 해결한다"
- "Apache Airflow의 execution_date, 멱등성, 백필 개념만 빌려온다"
- "Windows Server 단일 프로세스로 돌아가야 한다"

이 부분은 **여전히 사람의 영역**입니다. 도메인 지식(MES 환경, Oracle 11g 제약, 운영팀 역량)은 AI가 스스로 파악하기 어렵습니다. `docs/requirements.md`와 `docs/architecture.md`를 사람이 먼저 잡은 것처럼요.

---

### 2단계: AI가 모듈별 코드 초안을 대량 작성한다

**GPT-5.3에서**: 기존 모델이 학습 파이프라인, 데이터 전처리, 평가 코드 초안 작성

**airflow-lite에 대응하면**:
- 사람이 `PipelineRunner`의 인터페이스와 상태 전이 규칙을 설계하면
- AI가 `engine/pipeline.py`, `engine/state_machine.py`, `engine/strategy.py` 초안을 작성
- `storage/repository.py`의 CRUD 패턴, `extract/chunked_reader.py`의 제너레이터 패턴 같은 **반복적이지만 정확해야 하는 코드**를 AI가 빠르게 만듦
- 사람은 "이 Strategy 패턴이 맞는지", "WAL 모드가 이 환경에서 정말 필요한지" 같은 **아키텍처 판단**에 집중

현재 airflow-lite의 코드 구조를 보면, 모듈 간 패턴이 일관적입니다(ABC 패턴, dataclass 모델, repository CRUD). 이런 일관된 패턴의 반복 구현은 AI가 가장 잘하는 영역이죠.

---

### 3단계: AI가 프레임워크 자체를 써서 디버깅한다

**GPT-5.3에서**: 초기 버전이 자기 자신의 훈련을 디버깅하고 배포를 관리

**이게 가장 흥미로운 부분입니다.** airflow-lite에 대응하면:

- AI에게 실제로 airflow-lite를 **사용하게 합니다**
  - "이 YAML 설정으로 파이프라인을 등록하고 실행해 봐"
  - "API 엔드포인트를 호출해서 백필을 돌려 봐"
  - "일부러 Oracle 연결을 끊었을 때 상태 머신이 제대로 FAILED → PENDING 전이하는지 확인해 봐"
- AI가 **자기가 짠 코드를 직접 돌려보면서** 에러 로그를 읽고, 원인을 추정하고, 수정안을 제안
- 특히 `StageStateMachine`의 전이 규칙이나 `ChunkedReader`의 메모리 사용량 같은 건 **실행해봐야 버그가 드러나는** 영역

GPT-5.3이 "자기 훈련을 디버깅했다"는 건 결국 이겁니다: **자기가 만든 시스템을 자기가 테스트해서 문제를 찾는 피드백 루프**.

---

### 4단계: AI가 테스트와 엣지 케이스를 대량 생성한다

**GPT-5.3에서**: 기존 모델이 어려운 테스트 문제, 악의적 프롬프트, 함정 질문을 대량 생성

**airflow-lite에 대응하면**:
- AI가 `tests/` 아래에 테스트 케이스를 대량 생성:
  - **정상 경로**: 풀 마이그레이션 성공, 인크리멘탈 마이그레이션 성공
  - **실패 경로**: Oracle 연결 끊김 중간에, Parquet 쓰기 도중 디스크 풀, 동시에 같은 execution_date로 2번 실행
  - **엣지 케이스**: 빈 테이블, 컬럼이 전부 NULL, 1000만 행 테이블, execution_date가 미래 날짜
  - **상태 머신 퍼징**: 가능한 모든 상태 전이 조합을 시도해서 `InvalidTransitionError`가 정확히 나는지 확인
- 현재 `tests/` 디렉토리에 10개+ 테스트 모듈이 있는데, AI가 "이 테스트에서 빠진 시나리오"를 찾아서 추가 생성

사람은 "이 테스트가 실제로 의미 있는지" 필터링만 하면 됩니다.

---

### 5단계: AI가 설정 예시와 문서를 생성한다

**GPT-5.3에서**: 합성 데이터(좋은 답변 예시, 도구 호출 예시) 대량 생성

**airflow-lite에 대응하면**:
- `config/pipelines.yaml`의 다양한 설정 변형을 AI가 생성:
  - 테이블 10개짜리 설정
  - 인크리멘탈 + 풀 혼합 설정
  - 스케줄이 겹치는 설정 (충돌 테스트용)
- API 사용 예시 (curl 명령, Python 클라이언트 코드)
- 운영 가이드 ("Oracle 연결이 끊겼을 때 어떻게 복구하는가")
- 에러 메시지 → 해결 방법 매핑 문서

이건 "사람이 직접 다 쓰기엔 너무 많지만, 있으면 매우 유용한 것들"입니다.

---

### 6단계: AI가 성능 분석과 실험 결과를 읽는다

**GPT-5.3에서**: 수백 개 실험 결과를 읽고 "어느 설정이 좋았는지" 요약

**airflow-lite에 대응하면**:
- 다양한 `chunk_size` (5000, 10000, 20000, 50000)로 실행한 결과를 AI가 분석:
  - "chunk_size=20000일 때 메모리 450MB, 처리 시간 12분"
  - "chunk_size=50000일 때 메모리 520MB로 **제한 초과**, 처리 시간 9분"
  - "결론: 20000이 500MB 제한 안에서 최적"
- Snappy vs Zstd vs LZ4 압축 비교 결과 요약
- 동시 파이프라인 수에 따른 SQLite WAL 성능 변화 분석
- APScheduler의 misfire_grace_time 설정별 동작 차이 분석

사람 연구원이 하나하나 표를 만들 필요 없이, AI가 **1차 분석가** 역할을 합니다.

---

### 7단계: AI가 배포와 운영 체크를 돕는다

**GPT-5.3에서**: 초기 버전이 배포 관리를 도움

**airflow-lite에 대응하면**:
- Windows Service 설치 스크립트 검증
- `pyproject.toml` 의존성 버전 충돌 탐지
- 운영 환경과 개발 환경의 설정 차이 비교
- "이 설정으로 서비스를 시작하면 어떤 문제가 생길 수 있는지" 사전 점검
- 롤백 절차 문서화 (`.bak` 파일 복원 순서)

---

### 8단계: 보안과 안정성 검증

**GPT-5.3에서**: 시스템 카드에서 사이버 보안 보호장치를 별도 적용

**airflow-lite에 대응하면**:
- AI가 코드를 읽고 보안 취약점을 찾음:
  - "YAML 설정에 `${ORACLE_PASSWORD}`가 환경변수로 들어가는데, 로그에 찍히지 않는지?"
  - "API에 인증이 없는데, CORS만으로 충분한지?"
  - "SQL 인젝션 가능성은? (SQLite 쿼리에 파라미터 바인딩 쓰고 있는지)"
- 에러 핸들링 빈틈 탐지:
  - "OracleClient에서 ORA-03113 외에 놓친 retryable 에러코드는?"
  - "ParquetWriter의 backup→write→verify 과정에서 verify 실패 시 .bak 복원이 정말 안전한지?"

## 핵심 차이: GPT-5.3 개발 vs 프레임워크 개발

| 관점 | GPT-5.3 개발 | 프레임워크 개발 |
|------|-------------|--------------|
| **피드백 루프** | 모델 출력 품질로 평가 | 테스트 통과 + 실제 실행 결과로 평가 |
| **"자기 자신을 쓴다"** | 모델이 자기 훈련을 디버깅 | 프레임워크가 자기 CI/CD를 돌림 (셀프 호스팅 가능) |
| **합성 데이터** | 학습용 텍스트 데이터 | 테스트 픽스처, 설정 변형, 예시 문서 |
| **안전성** | 유해 응답 방지 | 데이터 유실 방지, 메모리 초과 방지 |

---

## 실제로 적용하려면 어떻게 시작하나

가장 현실적인 시작점:

1. **아키텍처와 인터페이스는 사람이 잡는다** — 이미 `docs/`에 잘 되어 있음
2. **AI에게 모듈 단위로 구현을 맡긴다** — 인터페이스 정의 + 기존 패턴 예시를 주고
3. **AI가 짠 코드를 AI가 테스트한다** — 테스트 케이스 생성 + 실행 + 실패 원인 분석
4. **사람은 리뷰어로 전환한다** — 코드 작성자가 아니라, 설계 판단자 + 최종 승인자

이게 바로 OpenAI가 말한 구조와 같습니다:

- **사람 = 총괄 설계자 + 최종 승인자**
- **AI = 코딩 어시스턴트 + 테스트 생성기 + 디버거 + 분석가**
