# AI-First Software Development Workflow

> AI 에이전트가 반복 구현을 주도하고 개발자가 설계, 리뷰, 승인을 담당하는 소프트웨어 개발 운영 문서
>
> 이 문서는 백엔드, API, CLI, UI, 배치, 라이브러리 등 **범용 소프트웨어 개발**에 적용한다. AI 모델 학습이나 정렬(alignment)은 이 문서의 범위가 아니다.
>
> 여기서 AI agent는 특정 벤더나 특정 모델에 종속되지 않는 추상적 실행 단위다. 에이전트 실행 인터페이스(현재 기본 구현체: Claude Code)는 교체 가능하며, 문서의 프로세스는 구현체에 독립적이다.

---

## 1. 문서 목표

이 문서의 목표는 소프트웨어 개발을 아래의 `반복형 개발 파이프라인`으로 운영하는 것이다.
아래 항목은 무작위 목록이 아니라, 기본 진행 순서를 나타내는 반복형 개발 루프다.

```text
요구사항 정의
  -> 수용 테스트 설계 (Acceptance Test First)
  -> 비기능 품질 기준 명세
  -> 작업 분해 및 코드베이스 인덱싱
  -> AI 구현 + 개발자 보완
  -> 자동 검증 및 자체 수정
  -> 비기능 품질 게이트 + 인간 리뷰
  -> 배포 게이트 + 지식 축적
```

실행 중에는 일부 단계가 반복해서 되돌아가지만, 기본 흐름은 위 순서를 따른다.
각 단계의 의미를 짧게 풀면 아래와 같다.

| 단계 | What | How |
|---|---|---|
| 요구사항 정의 | 이번 사이클에서 해결할 문제를 한 문장과 경계 조건으로 고정한다. | 요구사항, 장애, 개선 요청을 정리하고 성공 조건과 제외 범위를 명시한다. |
| 수용 테스트 설계 | 코드보다 테스트가 먼저다. 성공/실패를 자동 판정할 수 있는 테스트를 코드 구현 전에 설계한다. | 수용 테스트, 통합 테스트, 계약 테스트를 코드 작성 전에 정의한다. |
| 비기능 품질 기준 명세 | 기능 외 품질 속성(보안, 성능, 접근성, 호환성)의 기준을 선언적으로 고정한다. | 보안 스캔 기준, 성능 목표, 접근성 수준, 호환성 범위를 구조화된 형식으로 정리한다. |
| 작업 분해 및 코드베이스 인덱싱 | 큰 목표를 검증 가능한 작업으로 나누고, 코드베이스 구조를 기계적으로 파악한다. | 영향 범위, 쓰기 범위, 완료 정의, 병렬 가능 여부를 기준으로 task를 쪼갠다. |
| AI 구현 + 개발자 보완 | AI가 반복 구현을 담당하고, 개발자가 핵심 코드를 직접 작성하거나 보완한다. | task bundle을 입력으로 넣어 patch, 테스트, 작업 로그를 생성하게 한다. |
| 자동 검증 및 자체 수정 | 변경이 기본 품질 기준을 통과하는지 즉시 확인하고, 실패하면 자체 수정한다. | 테스트, 린트, 타입체크, 계약 검증, 시나리오 검증을 자동 실행하고 실패 시 수정 루프를 돌린다. |
| 비기능 품질 게이트 + 인간 리뷰 | 기능 검증과 독립된 품질 전용 검증을 실행하고, 개발자가 결과를 리뷰한다. | 보안 스캔, 성능 테스트, 접근성 검증을 실행하고, 개발자가 테스트 결과와 핵심 코드를 직접 검토한다. |
| 배포 게이트 + 지식 축적 | 결과를 종합해 배포 가능 여부를 판정하고, 이번 사이클의 교훈을 다음 사이클에 반영한다. | 모든 검증 통과, 품질 게이트 통과, 인간 승인 완료를 확인한다. 교훈을 규칙으로 축적한다. |

핵심은 세 가지다.

1. **개발자는 What/Why와 핵심 코드를 소유하고, AI는 반복 구현을 담당한다.** 개발자는 "무엇을 달성할 것인가"를 정의하고 아키텍처 경계, 보안 민감 로직, 복잡한 비즈니스 로직을 직접 작성한다. 반복 구현, 보일러플레이트, 테스트 작성은 AI가 담당한다.
2. **모든 판단은 자동화된 테스트로 측정한다.** 주관적 코드 리뷰만으로 판단하지 않고, 수용 테스트, 통합 테스트, 비기능 검증이 품질의 기준이 된다.
3. **매 사이클의 교훈이 다음 사이클의 규칙으로 축적된다.** 반복되는 실수, 승인된 패턴, 운영 사고 교훈을 자동으로 다음 사이클에 주입해 같은 실수를 줄인다.

## 2. 기본 원칙

이 워크플로는 특정 언어, 프레임워크, 도메인에 묶이지 않는다. 대신 아래 원칙을 공통 규칙으로 둔다.

### 2.1 개발자가 소유하는 것

- 요구사항 정의 (이 소프트웨어가 무엇을 해야 하는가)
- 수용 테스트 설계 (성공을 어떻게 측정할 것인가 — 테스트 기준은 개발자가 설계하고, AI는 테스트 코드만 구현한다)
- 아키텍처 방향 (시스템의 큰 구조를 결정하는 판단)
- 비기능 품질 기준 (보안, 성능, 접근성 목표)
- 리뷰와 승인 (모든 코드는 개발자가 승인해야 병합된다)
- **핵심 코드 직접 작성** — 아키텍처 경계, 보안 민감 로직, 복잡한 비즈니스 로직, AI가 반복 실패하는 영역

### 2.2 AI 에이전트에게 위임할 수 있는 것

- 반복 구현 (CRUD, 보일러플레이트, 패턴 기반 코드)
- 테스트 작성과 실행
- 버그 수정과 리팩토링
- AI-to-AI 코드 리뷰 (구현 에이전트와 별도의 리뷰 에이전트/도구)
- 문서 초안 작성
- 코드베이스 인덱싱과 분석

### 2.3 자동화 시스템이 소유하는 것

- CI/CD 파이프라인 실행
- 메트릭 수집과 대시보드
- 이상 탐지와 경보
- 배포 게이트 판정

### 2.4 프로젝트 제약은 하드코딩하지 않는다

이 문서는 어떤 프로젝트의 고유 제약도 전제로 하지 않는다.
대신 각 프로젝트는 별도의 `project-profile`을 이 워크플로에 주입한다.

예를 들어 프로젝트 프로파일에는 아래 같은 내용만 담는다.

- 런타임 환경
- 지원해야 하는 플랫폼
- 보안/규제 요구
- 성능 목표
- 호환성 규칙
- 배포 게이트 기준
- 금지된 변경 영역

즉, 워크플로는 고정이고 제약은 주입된다.

### 2.5 개발 도구와 개발 대상은 구분한다

Claude Code나 Copilot을 사용해 개발한다고 해서 결과물이 AI 제품일 필요는 없다.
이 문서에서 AI는 `개발 수행 수단`이며, 대상 프로젝트는 일반 소프트웨어 시스템 전체를 포함한다.

예를 들어 아래는 모두 이 문서의 대상이다.

- 배치 파이프라인
- 백엔드 API 서버
- 라이브러리 / SDK
- CLI 도구
- 프론트엔드 / UI
- 사내 업무 시스템

### 2.6 멀티 에이전트는 유용하지만 필수는 아니다

이 워크플로는 단일 에이전트로도 모든 단계가 작동한다.
멀티 에이전트는 규모가 커지거나 병렬 처리가 필요할 때 도입한다.
작업 성격에 따라 구현 agent, 리뷰 agent, 수정 agent를 분리할 수 있고,
같은 벤더의 모델을 역할별로 다르게 쓸 수도 있고, 서로 다른 AI agent를 조합할 수도 있다.

이 문서에서 이후 `AI`라는 표현은 단일 모델이 아니라, 작업에 따라 교체 가능한 하나 이상의 AI agent 집합을 뜻한다.

## 3. 필수 입력 산출물

이 워크플로를 자동화하려면 사람 문장만으로는 부족하다.
아래 산출물이 계속 갱신되어야 한다.
모든 구조화된 파일은 AI가 초안을 작성하되, 개발자가 승인한 것만 유효하다.

| 산출물 | 역할 |
|---|---|
| `project-profile.json` | 프로젝트별 제약, 금지사항, 비기능 목표 |
| `spec.json` | 이번 사이클에서 해결할 요구사항과 수용 기준 |
| `acceptance-tests/` | 코드보다 먼저 설계하는 수용 테스트 |
| `quality-constraints.yaml` | 보안, 성능, 접근성, 호환성 기준 |
| `tasks/*.yaml` | AI 에이전트가 처리할 작업 단위 — 역할, 쓰기 범위, handoff 규칙 포함 |
| `bundles/*.yaml` | 특정 작업에 필요한 문맥 패키지와 에이전트 실행 지시 |
| `check-report.json` | 자동 검증 결과 |
| `failure-bundle.json` | 수정에 필요한 실패 정보 묶음 |
| `quality-report.json` | 비기능 검증 결과 (보안, 성능, 접근성) |
| `gate-report.json` | 배포 가능 여부 판정 |

## 4. 에이전트 인터페이스와 기반 시스템

### 4.1 에이전트 실행 인터페이스

에이전트 실행 인터페이스는 task bundle을 읽고 적절한 에이전트를 호출해 결과를 저장하는 추상 계층이다.

- **현재 기본 구현체**: Claude Code (Anthropic CLI)
- 이 문서의 프로세스는 구현체에 독립적이다. 다른 에이전트 런타임으로 교체해도 워크플로는 동일하다.
- 핵심 요구사항: task bundle 입력 → 코드 patch + 실행 로그 + 결과 출력

### 4.2 기반 시스템

자동화된 AI 개발 루프는 모델보다 `도구 계층`이 먼저다.
가능하면 모든 도구는 CLI와 JSON 출력을 지원한다.

| 우선순위 | 시스템명 | 목적 | 최소 입력 | 최소 출력 |
|---|---|---|---|---|
| 1 | `profile-compiler` | 프로젝트 문서를 `project-profile.json`으로 정규화 | 운영 문서, 설계 문서 | `project-profile.json` |
| 2 | `spec-compiler` | 요구사항과 수용 기준을 `spec.json`으로 변환 | 기획/이슈/문서 | `spec.json` |
| 3 | `acceptance-test-designer` | 수용 테스트를 코드보다 먼저 설계 | `spec.json`, `project-profile.json` | `acceptance-tests/` |
| 4 | `repo-indexer` | 저장소 구조와 심볼을 인덱싱 | 소스, 테스트, 설정 | `repo-index.json` |
| 5 | `task-slicer` | 큰 요구사항을 원자 작업으로 분해 | `spec.json`, `repo-index.json` | `tasks/*.yaml` |
| 6 | `context-bundler` | 특정 작업용 입력 패키지 생성 | `task.yaml` | `bundle.yaml` |
| 7 | `agent-runner` | AI 실행기. bundle을 읽고 지정된 agent를 호출해 구현 로그와 결과를 저장 | `bundle.yaml` | `result.json`, patch |
| 8 | `check-runner` | 테스트, 린트, 타입체크, 계약검증, 시나리오 실행 | 변경 파일 또는 `task-id` | `check-report.json` |
| 9 | `failure-bundler` | 실패한 실행 결과를 수정 패키지로 압축 | `check-report.json`, logs | `failure-bundle.json` |
| 10 | `quality-gate-runner` | 보안, 성능, 접근성, 호환성 점검 | profile, dataset, deploy config | `quality-report.json` |
| 11 | `code-reviewer` | AI-to-AI 코드 리뷰, 패턴 검증, 보안 점검 | patch, 코드 컨텍스트 | 리뷰 리포트, 수정 요청 |
| 12 | `deployment-gate` | 모든 결과를 모아 배포 판정 | reports, diff, profile | `gate-report.json` |
| 13 | `knowledge-accumulator` | 승인된 패턴과 실패 교훈을 규칙으로 축적 | reports, postmortem | `patterns/*.md`, `lessons/*.json` |

### 4.3 2026년 기준 현실 도구 매핑

이 문서는 벤더 중립 추상화를 유지하지만, 실제 팀 도입 시에는 아래처럼 널리 쓰이는 도구 조합으로 구현하는 경우가 많다.
아래 목록은 `2026-03` 기준 공개 자료가 있는 경우는 그 자료를 따르고, 없는 경우는 공식 문서와 생태계 채택 신호를 바탕으로 고른 `대표 예시`다.

| 운영 항목 | 대표 도구 예시 (2026) | 이 문서의 추상 계층 | 메모 |
|---|---|---|---|
| 코딩 에이전트 실행 | `GitHub Copilot coding agent`, `Claude Code`, `Cursor Agent` | `agent-runner`, Stage 4 | 구현, 수정, PR 생성까지 직접 수행하는 실행 계층 |
| 팀 규칙 / 온보딩 | `CLAUDE.md`, `Cursor Rules`, `GitHub Copilot custom instructions` | `project-profile`, `spec`, Stage 0-2 | AI가 문제를 임의 재정의하지 못하게 만드는 상위 규칙 계층 |
| 반복 명령 / 작업 템플릿 | `Claude Code slash commands`, `GitHub Copilot prompt files`, `Cursor Custom Modes` | `spec-compiler`, `context-bundler`, Stage 0-3 | 자주 쓰는 작업을 재사용 가능한 명령/프롬프트로 고정 |
| 외부 도구 연결 | `MCP`, `GitHub MCP Server`, `Playwright MCP` | `context-bundler`, `agent-runner` | 저장소, 브라우저, 외부 시스템을 AI가 직접 다루게 하는 연결 계층 |
| 이슈 / 백로그 관리 | `Jira`, `GitHub Issues + Projects`, `Linear` | `spec-compiler`, `task-slicer` | 요구사항을 task로 구조화하고 우선순위를 관리하는 계층 |
| 품질 가드레일 / Hook | `Claude Code Hooks`, `pre-commit`, `Husky` | `check-runner`, `quality-gate-runner`, Stage 5-6 | 실수 방지, 규칙 강제, 커밋 전 자동 검사에 자주 사용 |
| PR / 코드리뷰 / CI | `GitHub Actions`, `CodeRabbit`, `SonarQube` | `check-runner`, `code-reviewer` | 자동 검증, PR 리뷰, 병합 전 차단 규칙에 주로 사용 |
| 브라우저 / E2E / 시나리오 검증 | `Playwright`, `Cypress`, `Selenium` | `check-runner` (시나리오) | unit test 밖의 실제 사용자 흐름 검증에 적합 |
| 보안 스캔 | `Snyk`, `Trivy`, `OWASP ZAP` | `quality-gate-runner` | 의존성, 컨테이너, 웹 애플리케이션 보안 점검 |
| 성능 테스트 | `k6`, `Locust`, `Lighthouse` | `quality-gate-runner` | 부하 테스트, 웹 성능 측정 |
| 접근성 테스트 | `axe-core`, `Lighthouse Accessibility` | `quality-gate-runner` | WCAG 준수 여부 자동 검증 |
| 인프라 / IaC / 배포 | `Terraform`, `OpenTofu`, `Pulumi` | `project-profile`, `deployment-gate` | 인프라 변경을 코드와 검증 파이프라인 안으로 넣는 계층 |
| 운영 관측 / 장애 추적 | `Grafana + Prometheus`, `Sentry`, `Langfuse` | `quality-gate-runner`, `knowledge-accumulator` | 서비스 운영 지표, 오류, AI 호출 품질을 추적하는 계층 |
| 멀티 에이전트 / 오케스트레이션 | `Claude Code parallel sessions`, `LangGraph`, `Ollama` | `agent-runner` (선택적) | 여러 agent를 묶거나 로컬/사내 실행 환경을 구성할 때 사용. 필수 아님 |

#### 도구 표를 읽는 방법

- 이 문서의 `profile-compiler`, `task-slicer`, `context-bundler` 같은 이름은 "반드시 그 제품을 써라"가 아니라 "그 역할을 수행하는 계층이 필요하다"는 뜻이다.
- 예를 들어 규칙 계층은 `CLAUDE.md` 하나로 끝나는 것이 아니라, `CLAUDE.md` + `Hooks` + `prompt files` + 저장소 규칙 문서 조합으로 운영되는 경우가 많다.
- GitHub 중심 조직이면 `GitHub Copilot` + `GitHub Actions` + `GitHub Issues` + `CodeRabbit` 조합이 자연스럽고, Claude 중심 조직이면 `Claude Code` + `MCP` + `Playwright MCP` + `CLAUDE.md` 조합이 자연스럽다.
- 따라서 실제 도입에서는 "도구명"보다 "어떤 추상 계층을 어떤 조합으로 채우는가"를 먼저 결정하는 편이 안전하다.

### 4.4 멀티 에이전트 운용 규칙

멀티 에이전트를 도입할 경우 아래 규칙을 따른다.
단일 에이전트로도 이 워크플로의 모든 단계가 작동한다.

각 task에는 아래 속성을 포함할 수 있다:

| 속성 | 설명 |
|---|---|
| `assigned_agent` | 이 task를 담당하는 에이전트 식별자 |
| `role` | 에이전트의 역할 — `implementer`, `reviewer`, `tester` 등 |
| `write_scope` | 이 에이전트가 수정 가능한 파일/디렉토리 목록 |
| `handoff` | 작업 완료 후 결과를 전달할 다음 에이전트 |
| `merge_policy` | 병렬 작업 결과의 병합 전략 — `sequential`, `independent`, `conflict-resolve` |

에이전트 간 운용 규칙:

- 쓰기 범위가 겹치는 task는 동시에 실행하지 않거나, `conflict-resolve` 병합 정책을 적용한다.
- 구현 에이전트와 리뷰 에이전트는 반드시 분리한다 (자기 코드를 자기가 리뷰하지 않는다).
- handoff는 구조화된 산출물(`task`, `bundle`, `result`)로만 한다.
- 에이전트 간 자연어 대화가 아닌 정형 데이터 교환을 원칙으로 한다.

## 5. 단계별 워크플로

아래 단계는 범용 소프트웨어 개발의 기본 순서다.
프로젝트마다 세부 체크 항목만 바뀌고 뼈대는 그대로 유지한다.

### Stage 0. 요구사항 정의 및 운영 계약 고정

#### 목적

이번 사이클에서 해결할 문제를 고정한다.
AI가 문제를 임의로 재정의하지 못하게 한다.

#### 개발자가 할 일

- 이번 사이클의 목표를 한 문장으로 고정한다.
- 성공 기준을 테스트 가능한 문장으로 바꾼다.
  - 예: "API 응답시간 p99 < 200ms", "결제 트랜잭션 성공률 > 99.9%", "모든 CRUD 엔드포인트에 대한 통합 테스트 통과"
- 이번 작업의 제외 범위를 적는다.
- 치명적 리스크를 명시한다.

#### AI가 할 일

- 모호한 표현을 표시한다.
- 누락된 승인 기준을 제안한다.
- 측정 가능한 acceptance criteria 초안을 만든다.
- `spec.json` 초안을 작성한다.

#### 산출물

- `project-profile.json`
- `spec.json`

#### 완료 기준

- "무엇을 왜 하는가"가 구조화되어 있다.
- 나중에 자동 판정 가능한 승인 기준이 있다.
- 개발자가 최종 승인했다.

---

### Stage 1. 수용 테스트 설계 — Acceptance Test First Development

#### 목적

코드보다 테스트가 먼저다.
요구사항을 자동으로 검증할 수 있는 테스트를 코드 구현 전에 설계한다.
이것이 이 워크플로의 핵심 차별점이다 — 코드를 먼저 쓰고 검증하는 것이 아니라, 테스트를 먼저 만들고 그 테스트를 통과하는 코드를 쓴다.

#### 개발자가 할 일

- 각 요구사항에 대한 테스트 방법을 결정한다.
  - 수용 테스트: 사용자 관점에서 기능이 올바르게 동작하는지
  - 통합 테스트: 모듈 간 연동이 올바른지
  - 계약 테스트: API/인터페이스가 명세를 준수하는지
- 테스트 데이터(fixture)의 출처와 품질 기준을 정한다.
- 통과/실패 기준을 수치로 고정한다.
- 회귀 탐지 규칙을 정의한다 (이전 버전 대비 성능 X% 이상 하락 시 실패).

#### AI가 할 일

- 테스트 코드를 작성한다.
- 테스트 데이터(fixture) 전처리 코드를 작성한다.
- 테스트 실행 인프라를 구축한다.
- 테스트가 의도한 기준을 올바르게 측정하는지 자체 검증 테스트를 작성한다.

#### 산출물

- `acceptance-tests/`
- 테스트 실행 코드
- 테스트 데이터 전처리 코드
- 테스트 자체 검증 코드

#### 완료 기준

- 모든 요구사항에 대응하는 테스트가 존재한다.
- 테스트가 자체 검증을 통과한다.
- 개발자가 테스트 설계를 승인했다 (올바른 것을 측정하는지 확인).

---

### Stage 2. 비기능 품질 기준 명세

#### 목적

기능 외 품질 속성의 기준을 선언적으로 고정한다.
비기능 품질은 기능 검증의 하위 항목이 아니라, 독립된 최상위 관심사다.

#### 개발자가 할 일

- 보안 기준을 명세한다.
  - 의존성 취약점 허용 수준 (critical/high는 0건)
  - 시크릿 노출 검사 범위
  - OWASP Top 10 준수 항목
- 성능 목표를 수치화한다.
  - latency (p50, p95, p99)
  - throughput (RPS)
  - 메모리/CPU 한계
  - 빌드 시간 한계
- 접근성 요구를 정의한다 (WCAG 2.1 AA 등).
- 호환성 범위를 명시한다 (브라우저, OS, 런타임 버전).
- 운영성 요구를 정의한다 (health check, graceful shutdown, 롤백).

#### AI가 할 일

- `quality-constraints.yaml` 초안을 작성한다.
- 각 기준에 대한 자동 검증 스크립트를 작성한다.
- 이전 버전의 품질 이슈 회귀 테스트를 작성한다.

#### 산출물

- `quality-constraints.yaml`
- 품질 검증 스크립트

#### 완료 기준

- 모든 비기능 기준이 자동 검증 가능한 수치/규칙을 가진다.
- 개발자가 품질 기준을 최종 승인했다.

---

### Stage 3. 작업 분해 및 코드베이스 인덱싱

#### 목적

큰 목표를 검증 가능한 작은 작업으로 나누고, 코드베이스 구조를 기계적으로 파악한다.
AI가 저장소를 추측하지 않게 한다.

#### 개발자가 할 일

- 큰 목표를 리뷰 가능한 묶음으로 나눈다.
- 우선순위와 의존관계를 정한다.
- 고위험 작업을 식별하고 추가 검증 기준을 붙인다.
- 핵심 모듈과 진입점을 표시한다.
- 변경 금지 영역을 지정한다.

#### AI가 할 일

- 저장소 구조를 분석하고 인덱싱한다.
- 최소 수정 범위를 가진 task 분해안을 제안한다.
- 각 task에 쓰기 범위, handoff 규칙을 배정한다.
- 병렬 실행 가능한 task 그룹을 식별한다.
- 쓰기 범위 충돌을 탐지하고 병합 정책을 제안한다.
- 사람이 놓친 선행조건을 표시한다.
- 재사용 가능한 기존 패턴을 찾는다.

#### task 예시

```yaml
task_id: TASK-042
title: "결제 API validation 강화"
assigned_agent: impl-agent-01
role: implementer
write_scope:
  - src/payment/api.py
  - src/payment/validators.py
  - tests/test_payment_api.py
must_pass:
  - tests/test_payment_api.py
  - acceptance-tests/payment_flow.py
handoff:
  - target: review-agent-01
    type: code-review
  - target: scenario-agent-01
    type: scenario-validation
merge_policy: independent
done_when:
  - "잘못된 결제 요청이 명확한 에러 메시지와 함께 거부된다"
  - "기존 API 시그니처가 유지된다"
  - "기존 테스트가 모두 통과한다"
```

#### 산출물

- `tasks/*.yaml`
- `bundles/*.yaml`
- `repo-index.json`
- `dependency-graph.json`

#### 완료 기준

- 각 작업이 작고 리뷰 가능하다.
- 각 작업마다 명확한 검증 경로가 있다.
- 쓰기 범위가 겹치는 task 쌍이 식별되고 병합 정책이 지정되었다.
- 관련 파일과 관련 테스트가 추측이 아니라 인덱스로 설명된다.

---

### Stage 4. AI 구현 + 개발자 보완

#### 목적

AI가 반복 구현을 담당하고, 개발자가 핵심 코드를 직접 작성하거나 보완한다.

#### 개발자가 할 일

- 고위험 작업에 아키텍처 방향이나 설계 힌트를 제공한다.
- execution profile과 사용할 agent를 선택한다.
- 자동 재시도 한계를 정한다.
- **직접 코드를 작성하는 영역:**
  - 아키텍처 경계가 바뀌는 코드
  - 보안 민감 로직 (인증, 권한, 암호화)
  - 복잡한 비즈니스 로직 (도메인 지식이 깊은 영역)
  - AI가 반복 실패하는 영역

#### AI가 할 일

- 각 에이전트가 자신의 bundle을 읽고 영향 범위를 확인한다.
- 기존 코드 패턴을 최대한 재사용한다.
- 쓰기 범위 내에서만 코드를 수정한다.
- 변경과 함께 테스트도 갱신한다.
- 구현 완료 후 handoff 대상에게 결과를 전달한다.

#### AI-to-AI 코드 리뷰

- 구현 에이전트와 별도의 리뷰 에이전트/도구(CodeRabbit, Claude `/review` 등)가 코드를 검토한다.
- 리뷰 에이전트가 제기한 이슈를 구현 에이전트가 반영한다.
- 이 단계는 인간 리뷰를 대체하는 것이 아니라, 보완하는 것이다.

#### 에이전트 간 충돌 해결

- **독립 영역**: `merge_policy: independent` — 병렬 실행 후 단순 병합
- **공유 영역**: `merge_policy: sequential` — 의존 순서대로 순차 실행
- **충돌 가능 영역**: `merge_policy: conflict-resolve` — 병렬 실행 후 충돌 해결 에이전트가 병합

#### 산출물

- 코드 patch (에이전트별)
- 변경 요약 (에이전트별)
- AI 생성 테스트
- handoff 기록

#### 완료 기준

- 각 에이전트의 수정이 task 범위를 넘지 않는다.
- 병합된 코드가 충돌 없이 통합된다.
- 로그만으로도 각 에이전트가 무엇을 했는지 추적 가능하다.

---

### Stage 5. 자동 검증 및 자체 수정 루프

#### 목적

Stage 1에서 설계한 수용 테스트와 기존 테스트로 구현 결과를 검증하고, 실패하면 자체 수정한다.
개별 테스트 통과와 별개로 실제 사용 흐름이 성립하는지도 확인한다.

#### 개발자가 할 일

- 필수 검증 세트를 정의한다.
- 무한 수정 루프를 막기 위한 재시도 횟수를 정한다.
- 사람 검토가 필요한 실패 유형을 구분한다.
- 대표 사용자 시나리오를 정의한다 (CLI 흐름, API 흐름, UI 흐름, 장애/복구 경로).

#### AI가 할 일

- 테스트, 린트, 타입체크를 실행한다.
- 시나리오 검증을 실행한다 (실제 명령, API, UI 흐름).
- 실패 원인을 로그 기반으로 분석한다.
- 수정 가능한 실패는 즉시 패치한다.
- 같은 실패가 반복되면 가설을 바꿔 다시 시도한다.
- 수정이 다른 테스트의 회귀를 일으키지 않는지 확인한다.

#### 절차

1. `check-runner`가 수용 테스트 + 단위 테스트 + 통합 테스트 + 린트 + 타입체크를 실행한다.
2. 실패 시 `failure-bundler`가 수정용 패키지를 만든다.
3. 수정 에이전트가 `failure-bundle.json`을 읽고 패치를 생성한다.
4. 다시 `check-runner`를 실행한다.
5. N회 반복 후에도 실패하면 개발자에게 에스컬레이션한다.
6. 기본 검증 통과 후, 시나리오 검증(CLI/API/UI 흐름)을 실행한다.
7. 시나리오 실패 시 같은 수정 루프를 적용한다.

#### 권장 정책

- 단순 테스트 실패: AI 자동 수정 2~3회
- 설계 충돌: 즉시 개발자 검토
- flaky 테스트 의심: 개발자 승인 필요
- 보안 경고: 자동 병합 금지, 개발자 판단
- 다른 테스트 회귀 발생: 자동 병합 금지, 개발자 판단

#### 산출물

- `check-report.json`
- `failure-bundle.json`
- `scenario-report.json`
- 수정된 patch

#### 완료 기준

- 수용 테스트와 기존 테스트가 모두 통과한다.
- 대표 시나리오가 실제로 동작한다.
- 이전 버전 대비 회귀가 없다.

---

### Stage 6. 비기능 품질 게이트 + 인간 리뷰

#### 목적

기능 검증(Stage 5)과 독립된 비기능 품질 전용 검증을 실행하고, 개발자가 전체 결과를 리뷰한다.
비기능 품질은 기능의 하위 항목이 아니라 별도의 최상위 게이트다 — 이 게이트를 통과하지 못하면 기능 테스트가 아무리 좋아도 배포할 수 없다.

#### 개발자가 할 일 — 품질 게이트

- 비기능 목표를 수치화한다.
- 허용 가능한 리스크와 불가 리스크를 구분한다.
- 품질 관련 예외 승인 여부를 결정한다.

#### AI가 할 일 — 품질 게이트

- Stage 2에서 정의한 품질 기준을 검증한다:
  - **보안**: 의존성 취약점 스캔, 시크릿 노출 검사, SAST/DAST
  - **성능**: 부하 테스트, latency 벤치마크, 메모리 프로파일링
  - **접근성**: WCAG 자동 검증
  - **호환성**: 브라우저/OS/런타임 호환성 테스트
  - **운영성**: health check, graceful shutdown, startup 검증
- 이전 버전 대비 품질 회귀를 탐지한다.
- 기준 미달 항목은 새 task로 환원한다.

#### 개발자가 할 일 — 리뷰

개발자는 아래를 검토한다:

1. **수용 테스트 결과 검토**: 모든 테스트가 통과했는가?
2. **비기능 품질 리포트 검토**: 보안, 성능, 접근성 기준을 충족하는가?
3. **아키텍처 변경 검토**: 시스템의 큰 구조가 의도대로 변경되었는가?
4. **AI-to-AI 리뷰 결과 검토**: 리뷰 에이전트가 제기한 이슈와 그 해결 과정.
5. **고위험 코드 직접 리뷰**: 보안 민감 코드, 아키텍처 경계 코드, 복잡한 비즈니스 로직은 개발자가 직접 코드를 읽는다.
6. **회귀 검토**: 이전 버전 대비 악화된 지표가 있는가?

개발자가 이슈를 발견하면 새로운 task로 AI에게 돌려보내거나, 직접 수정한다.

#### AI가 할 일 — 리뷰 지원

- 리뷰를 위한 요약 리포트를 생성한다.
- 테스트 결과, 품질 리포트, 아키텍처 변경 사항, 위험 항목을 한 문서에 정리한다.
- 개발자가 이해하기 쉬운 시각화를 만든다 (성능 차트, 비교표 등).
- 개발자가 지적한 이슈를 새 task로 변환한다.

#### 산출물

- `quality-report.json`
- 리뷰 요약 리포트
- 승인/반려 결정
- 반려 시: 새 task 목록

#### 완료 기준

- `quality-constraints.yaml`에 명시된 모든 기준을 충족한다.
- 개발자가 테스트 결과, 품질 리포트, 아키텍처 변경을 검토하고 명시적으로 승인했다.

---

### Stage 7. 배포 게이트 + 지식 축적

#### 목적

개발 완료와 배포 가능을 분리한다.
모든 검증, 품질 게이트, 인간 승인이 완료되었는지 최종 확인한다.
그리고 이번 사이클의 교훈을 다음 사이클에 반영한다.

#### 개발자가 할 일 — 배포 게이트

- 최종 체크리스트를 승인한다.
- 롤백 전략을 확인한다.
- 점진적 배포(canary/staged rollout) 계획을 승인한다.

#### AI가 할 일 — 배포 게이트

- 모든 리포트를 수집해 배포 리포트를 생성한다.
- 남은 리스크를 요약한다.
- 배포 스크립트와 롤백 스크립트를 작성한다.

#### 배포 게이트 최소 항목

- 모든 테스트 통과
- 대표 시나리오 통과
- 비기능 품질 게이트 통과
- 인간 리뷰 승인 완료
- 이전 버전 대비 회귀 없음
- 롤백 절차 존재
- 변경 요약과 리스크 문서 존재

#### 개발자가 할 일 — 지식 축적

- 실제 장애와 회귀를 기록한다.
- 자동화 규칙으로 승격할 리뷰 포인트를 고른다.
- 반복되는 실수를 공식 룰로 바꾼다 (예: CLAUDE.md에 규칙 추가).
- 다음 사이클의 요구사항을 설정한다.

#### AI가 할 일 — 지식 축적

- 운영 로그와 피드백을 분석한다.
- 공통 실패 패턴을 추출한다.
- 다음 사이클에 자동 주입할 규칙 초안을 만든다.

#### 축적해야 할 지식 종류

- 자주 깨지는 설계 규칙
- 자주 누락되는 테스트 유형
- 자주 발생하는 운영 사고 패턴
- 비기능 회귀 유발 패턴
- 프로젝트별 금지사항 위반 사례

#### 절차

1. `deployment-gate`가 모든 report와 diff를 수집한다.
2. 차단 항목을 명시적으로 판정한다.
3. 개발자가 최종 배포를 승인한다.
4. 배포 후 운영 이슈, 리뷰 이슈, 실패 보고서를 수집한다.
5. `knowledge-accumulator`가 공통 패턴을 규칙과 패턴 문서로 저장한다.
6. 이후 `context-bundler`가 새 bundle을 만들 때 이 규칙을 자동 포함한다.
7. Stage 0으로 돌아간다.

#### 산출물

- `gate-report.json`
- `deployment-plan.yaml`
- `rollback-guide.md`
- `patterns/*.md`
- `lessons/*.json`
- 차기 사이클용 rule set

#### 완료 기준

- "작동한다"가 아니라 "배포해도 된다" 수준까지 검증된다.
- 같은 실수를 반복할 확률이 줄어든다.
- 다음 사이클의 목표가 설정되었다.

## 6. 프로젝트 유형별 추가 게이트

범용 워크플로는 같지만 프로젝트 유형마다 강조점은 다르다.

### 라이브러리 / SDK

- public API snapshot 유지
- 예제 코드 컴파일/실행
- semver 호환성 확인
- 문서와 실제 시그니처 일치 여부 확인

### 백엔드 서비스 / API 서버

- API contract test
- startup/shutdown smoke
- migration/rollback 안전성
- health check와 장애 복구 검증

### 배치 / 데이터 파이프라인

- 재실행 가능성 (멱등성)
- 데이터 품질 기준
- partial failure 복구
- 자원 예산 준수

### CLI / 개발자 도구

- 주요 명령 시퀀스 검증
- 옵션 조합과 오류 메시지 확인
- 플랫폼별 동작 차이 점검
- backward compatible output 확인

### 프론트엔드 / UI

- 주요 사용자 플로우
- empty/loading/error state
- 접근성
- 시각 회귀

## 7. 실제 파이프라인 예시

아래는 "결제 모듈 리팩토링" 한 사이클의 구체적 흐름이다.
각 단계마다 역할(개발자/AI/자동)을 명시한다.

```text
 1. [개발자]  요구사항 정의 -> spec.json
 2. [AI]      spec-compiler가 수용 기준 정규화 -> spec.json 초안
 3. [개발자]  수용 기준 승인
 4. [AI]      acceptance-test-designer가 수용 테스트 설계 -> 테스트 코드
 5. [개발자]  테스트 설계 승인 (올바른 것을 측정하는가?)
 6. [개발자]  비기능 품질 기준 설정 -> quality-constraints.yaml
 7. [AI]      repo-indexer가 코드베이스 인덱싱 -> repo-index.json
 8. [AI]      task-slicer가 작업 분해 -> tasks/*.yaml
 9. [개발자]  작업 분해 승인
10. [AI]      구현 에이전트가 반복 구현 코드 작성 + 테스트 작성
11. [개발자]  보안 민감 로직(결제 검증, 암호화) 직접 작성
12. [AI]      리뷰 에이전트/CodeRabbit이 AI-to-AI 리뷰
13. [AI]      구현 에이전트가 리뷰 피드백 반영
14. [자동]    check-runner가 테스트/린트/타입체크 실행 -> check-report.json
15. [AI]      실패 시 수정 에이전트가 패치 -> 재검증 (최대 3회)
16. [자동]    시나리오 검증 (결제 API 흐름, 오류 응답, 재시도) -> scenario-report.json
17. [자동]    quality-gate-runner가 보안/성능 검증 -> quality-report.json
18. [개발자]  테스트 결과 + 품질 리포트 리뷰
19. [개발자]  보안 민감 코드 직접 리뷰
20. [개발자]  승인 또는 반려 (반려 시 새 task 생성 -> 10번으로)
21. [자동]    deployment-gate가 최종 판정 -> gate-report.json
22. [개발자]  배포 승인
23. [AI]      knowledge-accumulator가 교훈 축적 -> patterns/, lessons/
```

## 8. 전제 조건과 한계

### 8.1 전제 조건

이 워크플로가 작동하려면 아래 전제가 성립해야 한다.

| 전제 | 설명 |
|---|---|
| AI 코딩 품질이 실용적 수준 | AI가 생성하는 코드가 반복 구현, 패턴 기반 작업에서 실용적 수준이어야 한다. 모든 영역에서 인간을 초월할 필요는 없다. |
| 포괄적 테스트 체계 | AI 코드를 신뢰하려면 자동 테스트 커버리지가 높아야 한다. 테스트가 부족하면 AI가 생성한 코드의 품질을 보장할 수 없다. |
| 비기능 검증 자동화 | 보안, 성능, 접근성을 수동이 아닌 자동으로 검증할 수 있는 도구와 기준이 있어야 한다. |
| 충분한 API 비용 예산 | 반복적 AI 실행, 수정 루프, 리뷰에 API 호출 비용이 든다. |
| 버전 관리 인프라 | 코드, 테스트, 설정의 버전 관리와 롤백이 가능해야 한다. |

### 8.2 한계

| 한계 | 설명 | 완화 방법 |
|---|---|---|
| 테스트 커버리지 한계 | 테스트가 커버하지 못하는 영역은 AI 코드 품질을 보장할 수 없다. | 핵심 코드는 개발자가 직접 리뷰한다. 테스트 커버리지를 지속적으로 확장한다. |
| AI-to-AI 리뷰 맹점 | 같은 모델 계열의 리뷰어가 같은 방향으로 편향될 수 있다. | 서로 다른 도구 조합 (CodeRabbit + Claude `/review`)을 사용한다. 고위험 코드는 인간 리뷰를 병행한다. |
| 복잡한 비즈니스 로직 | 도메인 지식이 깊은 로직은 AI가 정확히 생성하기 어렵다. | 해당 영역은 개발자가 직접 구현하거나, 상세 명세와 예제를 제공한다. |
| 아키텍처 결정 | AI는 로컬 최적화에 강하지만 시스템 전체의 아키텍처 판단은 약하다. | 아키텍처 방향은 개발자가 결정하고, AI는 결정된 방향 안에서 구현만 담당한다. |
| 비용과 속도 | 반복적 AI 호출, 수정 루프에 API 비용과 시간이 든다. | 수정 루프 횟수를 제한한다. 비용 모니터링을 도입한다. 비용 대비 효과가 낮은 영역은 수동 전환한다. |

## 9. 도입 순서

처음부터 모든 단계를 자동화할 필요는 없다.
실무적으로는 아래 순서가 가장 안전하다.

1. **기본 자동화**: `profile-compiler`, `spec-compiler`, `repo-indexer`, `check-runner`를 먼저 만든다. 기존 CI/CD에 연결한다.
2. **AI 실행 루프**: `task-slicer`, `context-bundler`, `agent-runner`를 붙인다. AI가 task를 받아 구현하고 검증하는 기본 루프를 만든다.
3. **자체 수정 루프**: `failure-bundler`, `acceptance-test-designer`를 추가한다. 실패 시 AI가 자동 수정하는 루프를 완성한다.
4. **품질 게이트와 지식 축적**: `quality-gate-runner`, `deployment-gate`, `knowledge-accumulator`를 붙인다. 비기능 검증과 교훈 축적으로 사이클 전체를 완성한다.

단일 에이전트 + 기본 CI로 시작해서 점진적으로 확장하는 것이 가장 안전하다.

## 10. 이 문서를 실제로 쓰는 방법

새 기능, 리팩토링, 마이그레이션, 안정화 작업을 시작할 때마다 아래 순서로 쓴다.

1. `Stage 0`에서 요구사항과 수용 기준을 고정한다.
2. `Stage 1`에서 코드보다 먼저 수용 테스트를 설계한다.
3. `Stage 2`에서 비기능 품질 기준을 명세한다.
4. `Stage 3`에서 작업을 분해하고 코드베이스를 인덱싱한다.
5. `Stage 4`에서 AI가 구현하고 개발자가 보완한다.
6. `Stage 5`에서 자동 검증과 자체 수정 루프를 돌린다.
7. `Stage 6`에서 비기능 품질 게이트를 통과시키고 개발자가 리뷰한다.
8. `Stage 7`에서 배포 가능 여부를 판정하고 교훈을 축적한다.

이 구조가 자리 잡으면 개발자는 "모든 코드를 직접 쓰는 사람"에서 "요구사항을 구조화하고, 테스트를 설계하고, 품질을 보장하고, 핵심 코드만 직접 쓰는 사람"으로 이동한다. AI 에이전트 계층은 "가끔 도와주는 조수"가 아니라 "반복 구현과 검증을 담당하는 실행자"가 된다. 그리고 매 사이클의 교훈이 다음 사이클의 규칙으로 축적되어 팀 전체의 개발 효율이 점진적으로 향상된다.
