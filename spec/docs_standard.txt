하나의 문서 vs 분리된 문서
🔴 하나의 문서 — 언제 유리한가

프로젝트 규모가 작고 빠르게 프로토타이핑할 때
AI에게 "전체 맥락을 한 번에" 전달해야 할 때 (예: 첫 번째 대화 시작 시)
문서 간 참조가 너무 많아 분리하면 오히려 혼란스러울 때

단점: 문서가 길어질수록 AI의 집중도가 분산되고, 특정 섹션을 수정하면 나머지와 충돌 가능성 증가.

🟢 분리된 문서 — AI 입장에서 더 유리한 이유
이유설명역할 분리요구사항은 "무엇을", 설계는 "어떻게", 작업분해는 "순서와 단위"로 관심사가 다름컨텍스트 오염 방지AI가 설계 문서를 볼 때 요구사항 노이즈 없이 집중 가능갱신 충돌 최소화요구사항이 바뀌어도 설계 문서 전체를 재작성하지 않아도 됨에이전트 작업단위 독립성에이전트는 태스크 단위 문서만 읽고 실행하면 됨 — 요구사항까지 읽을 필요 없음토큰 효율필요한 문서만 컨텍스트에 주입 가능

실전 추천 구조
docs/
├── requirements.md      # 요구사항 (What)
├── architecture.md      # 설계 (How)
└── tasks/
    ├── task-001.md      # 에이전트 작업단위 (Do)
    ├── task-002.md
    └── ...
각 문서의 역할
requirements.md
→ AI가 "왜 이걸 만드는지" 이해하는 기준 문서. 변경 이력 관리 대상.
architecture.md
→ AI가 코드 생성 시 참조하는 설계 제약. 지원형(FastAPI/DuckDB 등) 스택 결정사항 포함.
→ 코드는 필요 시 인터페이스 수준으로 작성
tasks/
task-XXX.md
→ 에이전트가 실제로 실행할 때 주입되는 문서. 자기완결적으로 작성해야 함 — 다른 문서를 안 봐도 실행 가능한 수준.
→ 구현 레시피 (라이브러리 설정값, 데코레이터 파라미터, 미들웨어 설정 등), 구현 가이드 작성

핵심 원칙

에이전트 작업단위 문서는 반드시 자기완결적(self-contained)으로.

에이전트는 매 실행마다 컨텍스트가 초기화되기 때문에, task 문서 안에 "입력 / 출력 / 제약 / 완료 조건"이 명시되어 있어야 합니다. 이 부분이 요구사항/설계와 분리되어야 하는 가장 강한 이유예요.

실제로 어떻게 다른가
❌ 자기완결적이지 않은 task (나쁜 예)
markdown# Task-001: 마이그레이션 엔진 구현

requirements.md의 3.2절 참고.
architecture.md에 정의된 Strategy 패턴으로 구현할 것.
→ 에이전트가 두 문서를 못 받으면 아무것도 못 함.

✅ 자기완결적인 task (좋은 예)
markdown# Task-001: 마이그레이션 엔진 구현

## 목적
Oracle 11g의 월별 데이터를 Parquet 파일로 변환한다.

## 입력
- Oracle 연결: oracledb, 접속 정보는 config.py에서 로드
- 대상 테이블: PRODUCTION_LOG (파티션 기준: LOG_DATE)

## 출력
- 경로: /data/parquet/PRODUCTION_LOG/YYYY=MM/
- 포맷: Parquet, Snappy 압축

## 구현 제약
- Strategy 패턴 사용 (FullMigration, IncrementalMigration)
- 재실행 시 기존 파일 덮어쓰기 전 백업 필수
- Python / FastAPI 생태계 사용

## 완료 조건
- [ ] FullMigrationStrategy 클래스 구현
- [ ] IncrementalMigrationStrategy 클래스 구현
- [ ] 단위 테스트 통과
→ 이 문서 하나만으로 에이전트가 실행 가능.

단, 한 가지 현실적인 타협점
requirements/architecture의 **핵심 내용을 task에 복사(또는 요약 인라인)**하는 건 맞지만, 매번 전부 복붙하면 유지보수가 힘들어집니다.
그래서 실전에서는 이렇게 씁니다:
markdown## 기술 스택 (공통 제약)
- Python 3.11 / FastAPI / DuckDB / Parquet
- 이 task에서 변경 불가

참조의 두 가지 용도
용도설명task에 넣어도 되나?실행에 필요한 정보이걸 봐야 작업이 가능함❌ 넣으면 안 됨 — task 안에 직접 써야 함추가 맥락 / 배경없어도 작업 가능, 더 깊이 이해하고 싶을 때✅ 링크로 남겨도 됨

구체적으로
markdown# Task-001: 마이그레이션 엔진 구현

## 구현 제약
- Strategy 패턴 사용 (FullMigration, IncrementalMigration)
- 재실행 시 기존 파일 백업 필수

## 참고 (선택)
- 전체 아키텍처 배경: architecture.md
- 요구사항 원문: requirements.md
이렇게 "참고 (선택)" 섹션으로 맨 아래에 분리해두면, 에이전트는 위에서 아래로 읽다가 참고 섹션은 무시하고 실행할 수 있어요.

핵심 기준 한 줄

"이 링크가 없으면 작업을 못 하나?" → Yes면 링크 말고 내용을 직접 써라. No면 링크로 남겨도 된다.
