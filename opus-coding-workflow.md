# AI-Authored Coding Workflow

> Claude Opus 4.6을 포함한 차세대 AI 모델 개발에서, **사람이 코드를 한 줄도 쓰지 않는** 완전 자율 개발 워크플로를 설계하기 위한 운영 문서
>
> 이 문서는 AI 기능 제품(AI-powered product)을 만드는 문서가 아니다. **AI가 소프트웨어를 만드는 개발 프로세스** 문서다. 대상 소프트웨어가 AI 모델이든, 백엔드 서버든, CLI 도구든 동일하게 적용된다.
>
> 여기서 AI agent는 특정 벤더나 특정 모델에 종속되지 않는 추상적 실행 단위다. 에이전트 실행 인터페이스(현재 기본 구현체: Claude Code)는 교체 가능하며, 문서의 프로세스는 구현체에 독립적이다.

---

## 1. 문서 목표

이 문서의 목표는 AI 모델 개발을 아래의 `재귀적 개선 파이프라인`으로 운영하는 것이다.
아래 항목은 무작위 목록이 아니라, 기본 진행 순서를 나타내는 재귀형 개발 루프다.

```text
능력 목표 정의
  -> 평가 체계 설계
  -> 안전성 제약 명세
  -> 작업 분해 및 멀티 에이전트 배정
  -> 병렬 AI 코드 생성
  -> 자동 평가 및 자체 수정
  -> 안전성 검증 게이트
  -> 인간 리뷰 및 승인
  -> 배포 게이트
  -> 운영 피드백 + 재귀 루프 진입
```

실행 중에는 일부 단계가 반복해서 되돌아가지만, 기본 흐름은 위 순서를 따른다.
각 단계의 의미를 짧게 풀면 아래와 같다.

| 단계 | What | How |
|---|---|---|
| 능력 목표 정의 | 이번 사이클에서 달성해야 할 모델 능력을 정량 기준으로 고정한다. | 벤치마크 목표, 능력 영역, 안전성 기준을 구조화된 형식으로 정리한다. |
| 평가 체계 설계 | 코드보다 평가가 먼저다. 성공/실패를 자동 판정할 수 있는 eval suite를 설계한다. | 능력 목표마다 측정 방법, 데이터셋, 통과 기준, 회귀 탐지 규칙을 정의한다. |
| 안전성 제약 명세 | 모든 개발 단계에 적용될 안전성 제약을 선언적으로 고정한다. | Constitutional AI 원칙, 거부 기준, 편향 한계, 레드팀 시나리오를 YAML로 정리한다. |
| 작업 분해 및 멀티 에이전트 배정 | 큰 목표를 검증 가능한 작업으로 나누고, 각 작업에 담당 에이전트를 배정한다. | 영향 범위, 에이전트 역할, 쓰기 범위, handoff 규칙, 병합 정책을 기준으로 task를 쪼갠다. |
| 병렬 AI 코드 생성 | 다중 에이전트가 동시에 각자의 task를 구현한다. | task bundle을 입력으로 넣어 patch, 테스트, 작업 로그를 생성하게 한다. |
| 자동 평가 및 자체 수정 | 변경이 eval suite를 통과하는지 즉시 확인하고, 실패하면 자체 수정한다. | eval 실행, 실패 분석, 수정 patch 생성, 재평가를 자동 반복한다. |
| 안전성 검증 게이트 | 기능 검증과 독립된 안전성 전용 검증을 실행한다. | 레드팀 테스트, 편향 탐지, 거부 정확도, 탈옥 방어를 별도 게이트로 실행한다. |
| 인간 리뷰 및 승인 | 코드를 쓰지 않는 사람이 결과물을 리뷰하고 승인한다. | eval 결과, 안전성 리포트, 아키텍처 변경 요약, 위험 항목을 검토한다. |
| 배포 게이트 | 결과를 종합해 배포 가능 여부를 판정한다. | 모든 eval 통과, 안전성 게이트 통과, 인간 승인 완료를 확인한다. |
| 운영 피드백 + 재귀 루프 진입 | 배포 후 피드백을 수집하고, 새 버전이 다음 사이클의 worker로 승격된다. | 운영 메트릭, 사용자 피드백, 사고 보고서를 다음 사이클의 입력으로 주입한다. |

### 역할 비교표: codex-coding-workflow vs opus-coding-workflow

| 영역 | codex-coding-workflow | opus-coding-workflow |
|---|---|---|
| 사람의 역할 | 문제 정의 + **일부 코딩** + 리뷰 + 승인 | 문제 정의 + 평가 설계 + 리뷰 + 승인 (**코딩 0%**) |
| AI의 역할 | 반복 구현, 테스트 초안, 수정 | **모든 코드 작성** — 학습 인프라, 데이터 파이프라인, 평가 코드, 서빙 코드 전부 |
| 코드 소유권 | 사람이 최종 코드 소유 | AI가 작성, 사람은 **승인 권한**만 소유 |
| 재귀적 개선 | 없음 | Claude N이 Claude N+1을 만든다 |
| 안전성 | 비기능 검증의 일부 | **독립된 안전성 게이트** (Stage 6) |
| 평가 우선순위 | 코드 이후 검증 | **코드보다 평가가 먼저** (Eval-Driven Development) |
| 에이전트 구성 | 단일/복수 선택 가능 | **멀티 에이전트 병행이 기본** |
| 도구 제작 | 사람이 먼저 만든다 | **도구 자체도 AI가 작성** |

핵심은 세 가지다.

1. **사람은 What/Why만, How는 AI가 소유한다.** 사람은 "무엇을 달성할 것인가"와 "왜 이것이 중요한가"를 정의하고, "어떻게 구현할 것인가"는 전적으로 AI 에이전트가 결정한다.
2. **모든 판단은 eval로 측정한다.** 주관적 코드 리뷰가 아니라 자동화된 평가 체계가 품질의 기준이다.
3. **Claude N이 Claude N+1을 만든다.** 현재 버전의 모델이 다음 버전의 모델을 만드는 학습 인프라, 데이터 파이프라인, 평가 코드를 작성한다.

## 2. 기본 원칙 — codex-coding-workflow와의 결정적 차이

이 워크플로는 참조 문서(codex-coding-workflow)의 원칙을 공유하되, 아래에서 결정적으로 갈라진다.

### 2.1 사람이 소유하는 것

- 능력 목표 (이 모델이 무엇을 할 수 있어야 하는가)
- 평가 설계 (성공을 어떻게 측정할 것인가)
- 안전성 제약 (모델이 절대 해서는 안 되는 것)
- 아키텍처 방향 (시스템의 큰 구조를 결정하는 판단)
- 리뷰와 승인 (모든 코드는 사람이 승인해야 병합된다)

사람이 소유하지 **않는** 것: 코드 구현. 한 줄도 없다.

### 2.2 AI 에이전트가 소유하는 것

- **모든 코드 작성** — 학습 인프라, 데이터 파이프라인, 평가 코드, 서빙 코드, 모니터링 코드
- 테스트 작성과 실행
- 버그 수정과 최적화
- AI-to-AI 코드 리뷰 (구현 에이전트와 별도의 리뷰 에이전트)
- 문서 초안 작성
- 도구 자체의 구현 (참조 문서와의 차이: 참조 문서는 도구를 사람이 먼저 만든다고 전제)

### 2.3 자동화 시스템이 소유하는 것

- 평가(eval) 실행과 결과 수집
- CI/CD 파이프라인 실행
- 메트릭 수집과 대시보드
- 이상 탐지와 경보
- 배포 게이트 판정

### 2.4 참조 문서와의 비교

| 원칙 | codex-coding-workflow | opus-coding-workflow |
|---|---|---|
| 프로젝트 제약 주입 | project-profile로 주입 | capability-goal + safety-constraints + eval-suite로 주입 |
| 도구와 대상의 분리 | "AI는 개발 수행 수단" | "AI는 개발 수행 수단이자, 개발 대상이기도 하다" |
| 단일 agent 비전제 | 복수 agent 허용 | **멀티 에이전트 병행이 기본 전제** |
| 코드베이스 인덱싱 | Stage 1에서 수행 | AI가 인덱서 자체도 작성 |
| 안전성 | 비기능 검증에 포함 | 독립된 Stage로 분리 |

## 3. 재귀적 자기 개선의 구조

이 섹션은 참조 문서에 없는 핵심 차별점이다.

### 3.1 버전 체인

```text
Claude N (worker)
  -> 학습 인프라 코드 작성
  -> 데이터 파이프라인 코드 작성
  -> 평가 프레임워크 코드 작성
  -> 서빙 인프라 코드 작성
  -> 학습 실행
  -> 평가 실행
  -> 안전성 검증
  -> 인간 승인
  -> Claude N+1 (product)

Claude N+1이 다음 사이클의 worker로 승격
  -> Claude N+1 (worker)
  -> ...
  -> Claude N+2 (product)
```

- **Worker**: 코드를 작성하는 현재 버전의 모델
- **Product**: 작성된 코드로 학습된 다음 버전의 모델
- Worker와 product는 동일한 모델 계열이지만 버전이 다르다

### 3.2 자기 참조 위험 관리

재귀적 자기 개선에서 가장 위험한 것은 "자기가 자기를 평가하는" 순환이다. 이를 방지하기 위한 규칙:

1. **Eval 기준은 사람이 설계한다.** AI는 eval을 실행하는 코드만 구현한다. "무엇이 좋은 모델인가"의 정의는 사람에게 있다.
2. **Eval 코드의 정확성은 독립 검증한다.** eval 코드가 의도한 기준을 올바르게 측정하는지 사람이 확인한다.
3. **Worker는 자신의 후계 모델의 안전성 기준을 수정할 수 없다.** 안전성 제약은 사람이 설정하고, worker는 이를 구현만 한다.
4. **Eval 결과를 조작하는 코드 패턴을 자동 탐지한다.** eval score를 인위적으로 올리는 코드(reward hacking)를 정적/동적으로 검출한다.

### 3.3 버전 격리

Worker와 product의 엄격한 분리:

| 영역 | Worker (Claude N) | Product (Claude N+1) |
|---|---|---|
| 역할 | 코드 작성, 테스트, 수정 | 학습 대상, 평가 대상 |
| 접근 가능 범위 | 전체 코드베이스, 도구 체인 | 학습 데이터, 학습 설정 (코드 수정 불가) |
| 평가 주체 | Worker가 아닌 독립 eval 시스템 | 사람 + 자동 eval |
| 안전성 기준 수정 | 불가 (사람만 가능) | 해당 없음 |

## 4. 필수 입력 산출물

이 워크플로를 운영하려면 사람이 아래 산출물을 유지해야 한다.
모든 구조화된 파일은 AI가 초안을 작성하되, 사람이 승인한 것만 유효하다.

| 산출물 | 역할 |
|---|---|
| `capability-goal.yaml` | 이번 사이클에서 달성할 능력 목표와 정량 기준 |
| `safety-constraints.yaml` | 모델에 적용될 안전성 제약, 거부 기준, 편향 한계 |
| `eval-suite.yaml` | 평가 체계 — 벤치마크, 데이터셋, 통과 기준, 회귀 탐지 규칙 |
| `agent-tasks/*.yaml` | AI 에이전트가 처리할 작업 단위 — 역할, 쓰기 범위, handoff 규칙 포함 |
| `agent-bundles/*.yaml` | 특정 작업에 필요한 문맥 패키지와 에이전트 실행 지시 |
| `eval-report.json` | 자동 평가 결과 |
| `safety-report.json` | 안전성 검증 결과 |
| `failure-bundle.json` | 수정에 필요한 실패 정보 묶음 |
| `gate-report.json` | 배포 가능 여부 판정 |
| `version-chain.json` | worker/product 버전 이력과 승격 기록 |

## 5. 에이전트 인터페이스와 기반 시스템

### 5.1 에이전트 실행 인터페이스

에이전트 실행 인터페이스는 task bundle을 읽고 적절한 에이전트를 호출해 결과를 저장하는 추상 계층이다.

- **현재 기본 구현체**: Claude Code (Anthropic CLI)
- 이 문서의 프로세스는 구현체에 독립적이다. 다른 에이전트 런타임으로 교체해도 워크플로는 동일하다.
- 핵심 요구사항: task bundle 입력 → 코드 patch + 실행 로그 + 결과 출력

### 5.2 기반 시스템 10개

자동화된 AI 개발 루프는 모델보다 `시스템 계층`이 먼저다.
**참조 문서와의 핵심 차이: 이 기반 시스템 자체도 AI가 작성한다.**
사람은 각 시스템의 요구사항과 인터페이스를 정의하고, AI가 구현하며, 사람이 승인한다.

| 우선순위 | 시스템명 | 목적 | 최소 입력 | 최소 출력 |
|---|---|---|---|---|
| 1 | `goal-compiler` | 능력 목표를 구조화된 형식으로 정규화 | 연구 문서, 벤치마크 목록 | `capability-goal.yaml` |
| 2 | `eval-designer` | 능력 목표마다 평가 방법, 데이터셋, 통과 기준 설계 | `capability-goal.yaml` | `eval-suite.yaml` |
| 3 | `safety-constraint-compiler` | 안전성 제약을 선언적 형식으로 정규화 | 정책 문서, Constitutional AI 원칙 | `safety-constraints.yaml` |
| 4 | `agent-orchestrator` | 작업 분해, 에이전트 배정, 병렬 실행 관리 | goal, eval, constraints, repo | `agent-tasks/*.yaml`, 실행 스케줄 |
| 5 | `training-pipeline-runner` | 학습 인프라 코드 실행, 체크포인트 관리 | 학습 설정, 데이터 경로 | 체크포인트, 학습 로그 |
| 6 | `eval-runner` | eval suite를 실행하고 결과 수집 | `eval-suite.yaml`, 모델 체크포인트 | `eval-report.json` |
| 7 | `safety-tester` | 안전성 전용 검증 — 레드팀, 편향, 거부 정확도 | `safety-constraints.yaml`, 모델 | `safety-report.json` |
| 8 | `code-reviewer` | AI-to-AI 코드 리뷰, 패턴 검증, 보안 점검 | patch, 코드 컨텍스트 | 리뷰 리포트, 수정 요청 |
| 9 | `deployment-gate` | 모든 결과를 모아 배포 판정 | reports, diff, constraints | `gate-report.json` |
| 10 | `knowledge-accumulator` | 승인된 패턴과 실패 교훈을 규칙으로 축적 | reports, postmortem | `patterns/*.md`, `lessons/*.json` |

### 5.3 멀티 에이전트 운용 규칙

멀티 에이전트 병행은 이 워크플로의 기본 전제다. 단일 에이전트 루프는 특수 케이스로 취급한다.

각 task와 bundle에는 아래 속성이 필수로 포함된다:

| 속성 | 설명 |
|---|---|
| `assigned_agent` | 이 task를 담당하는 에이전트 식별자 |
| `role` | 에이전트의 역할 — `implementer`, `reviewer`, `tester`, `safety-auditor` 등 |
| `write_scope` | 이 에이전트가 수정 가능한 파일/디렉토리 목록 |
| `handoff` | 작업 완료 후 결과를 전달할 다음 에이전트 |
| `merge_policy` | 병렬 작업 결과의 병합 전략 — `sequential`, `independent`, `conflict-resolve` |

에이전트 간 운용 규칙:

- 쓰기 범위가 겹치는 task는 동시에 실행하지 않거나, `conflict-resolve` 병합 정책을 적용한다.
- 구현 에이전트와 리뷰 에이전트는 반드시 분리한다 (자기 코드를 자기가 리뷰하지 않는다).
- handoff는 구조화된 산출물(`task`, `bundle`, `result`)로만 한다.
- 에이전트 간 자연어 대화가 아닌 정형 데이터 교환을 원칙으로 한다.

---

## 6. 단계별 워크플로

아래 단계는 재귀적 AI 모델 개발의 범용 순서다.
프로젝트마다 세부 체크 항목만 바뀌고 뼈대는 그대로 유지한다.

### Stage 0. 능력 목표 고정

#### 목적

이번 사이클에서 달성해야 할 모델 능력을 정량 기준으로 고정한다.
AI가 목표를 임의로 재정의하지 못하게 한다.

#### 사람이 할 일

- 이번 사이클의 능력 목표를 정량 기준으로 고정한다.
  - 예: "HumanEval pass@1 > 92%", "MMLU > 90%", "코딩 벤치마크 X에서 Y 이상"
- 목표 간 우선순위를 정한다.
- 이번 사이클의 제외 범위를 적는다.
- 치명적 리스크를 명시한다.

#### AI가 할 일

- 모호한 목표를 표시한다.
- 누락된 측정 기준을 제안한다.
- 목표 간 충돌을 탐지한다.
- `capability-goal.yaml` 초안을 작성한다.

#### 산출물

- `capability-goal.yaml`

#### 완료 기준

- 모든 능력 목표가 자동 측정 가능한 정량 기준을 가진다.
- 사람이 최종 승인했다.

---

### Stage 1. 평가 체계 설계 — Eval-Driven Development

#### 목적

코드보다 평가가 먼저다.
능력 목표를 자동으로 측정할 수 있는 eval suite를 코드 구현 전에 설계한다.
이것이 참조 문서와의 근본적 차이다 — 참조 문서는 코드를 먼저 쓰고 검증하지만, 이 워크플로는 평가 체계를 먼저 만들고 그 평가를 통과하는 코드를 쓴다.

#### 사람이 할 일

- 각 능력 목표에 대한 평가 방법을 결정한다.
- 평가 데이터셋의 출처와 품질 기준을 정한다.
- 통과/실패 기준을 수치로 고정한다.
- 회귀 탐지 규칙을 정의한다 (이전 버전 대비 X% 이상 하락 시 실패).

#### AI가 할 일

- eval 코드를 작성한다.
- 평가 데이터셋 전처리 코드를 작성한다.
- 평가 실행 인프라를 구축한다.
- eval 코드가 의도한 기준을 올바르게 측정하는지 자체 검증 테스트를 작성한다.
- `eval-suite.yaml` 초안을 작성한다.

#### 산출물

- `eval-suite.yaml`
- eval 실행 코드
- 평가 데이터셋 전처리 코드
- eval 자체 검증 테스트

#### 완료 기준

- 모든 능력 목표에 대응하는 eval이 존재한다.
- eval 코드가 자체 검증 테스트를 통과한다.
- 사람이 eval 설계를 승인했다 (eval이 올바른 것을 측정하는지 확인).

---

### Stage 2. 안전성 제약 명세

#### 목적

모든 개발 단계에 적용될 안전성 제약을 선언적으로 고정한다.
안전성은 비기능 요구사항의 하위 항목이 아니라, 독립된 최상위 제약이다.

#### 사람이 할 일

- Constitutional AI 원칙을 정의하거나 갱신한다.
- 모델의 거부 기준을 명세한다 (무엇을 거부해야 하고, 무엇을 거부하면 안 되는가).
- 편향 한계를 수치화한다.
- 레드팀 시나리오를 설계한다.
- 안전성 게이트의 통과 기준을 고정한다.

#### AI가 할 일

- `safety-constraints.yaml` 초안을 작성한다.
- 안전성 테스트 코드를 작성한다.
- 레드팀 시나리오 실행 코드를 작성한다.
- 편향 탐지 코드를 작성한다.
- 이전 버전에서 발견된 안전성 이슈의 회귀 테스트를 작성한다.

#### 산출물

- `safety-constraints.yaml`
- 안전성 테스트 코드
- 레드팀 시나리오 실행 코드
- 편향 탐지 코드

#### 완료 기준

- 모든 안전성 제약이 자동 검증 가능한 테스트를 가진다.
- 사람이 안전성 기준을 최종 승인했다.

---

### Stage 3. 작업 분해 및 멀티 에이전트 배정

#### 목적

큰 목표를 검증 가능한 작은 작업으로 나누고, 각 작업에 담당 에이전트를 배정한다.
다중 에이전트가 충돌 없이 병렬로 작업할 수 있는 구조를 만든다.

#### 사람이 할 일

- 큰 목표를 리뷰 가능한 묶음으로 나눈다.
- 우선순위와 의존관계를 정한다.
- 고위험 작업을 식별하고 추가 검증 기준을 붙인다.

#### AI가 할 일

- 최소 수정 범위를 가진 task 분해안을 제안한다.
- 각 task에 에이전트 역할, 쓰기 범위, handoff 규칙을 배정한다.
- 병렬 실행 가능한 task 그룹을 식별한다.
- 쓰기 범위 충돌을 탐지하고 병합 정책을 제안한다.
- 사람이 놓친 선행조건을 표시한다.

#### task 예시

```yaml
task_id: TASK-017
title: "사전학습 데이터 로더 최적화"
assigned_agent: impl-agent-03
role: implementer
write_scope:
  - src/training/data_loader.py
  - src/training/data_pipeline.py
  - tests/test_data_loader.py
must_pass:
  - tests/test_data_loader.py
  - evals/throughput_benchmark.py
handoff:
  - target: review-agent-01
    type: code-review
  - target: perf-agent-01
    type: performance-validation
merge_policy: independent
done_when:
  - "데이터 로딩 throughput이 기존 대비 20% 이상 향상"
  - "기존 테스트가 모두 통과"
  - "메모리 사용량이 기존 대비 증가하지 않음"
```

#### 산출물

- `agent-tasks/*.yaml`
- `dependency-graph.json`
- `parallel-execution-plan.json`

#### 완료 기준

- 각 작업이 작고 리뷰 가능하다.
- 각 작업마다 명확한 에이전트 배정과 검증 경로가 있다.
- 쓰기 범위가 겹치는 task 쌍이 식별되고 병합 정책이 지정되었다.

---

### Stage 4. 병렬 AI 코드 생성

#### 목적

다중 에이전트가 동시에 각자의 task를 구현한다.
사람은 코드를 쓰지 않는다. 사람은 고위험 작업에 대한 아키텍처 힌트만 제공한다.

#### 사람이 할 일

- 고위험 작업에 아키텍처 방향이나 설계 힌트를 제공한다.
- 에이전트별 실행 프로파일(execution profile)을 선택한다.
  <!-- 모델 고유 개념 참고: Anthropic의 경우 extended thinking 활성화 여부 등 -->
- 자동 재시도 한계를 정한다.

#### AI가 할 일

- 각 에이전트가 자신의 bundle을 읽고 영향 범위를 확인한다.
- 기존 코드 패턴을 최대한 재사용한다.
- 쓰기 범위 내에서만 코드를 수정한다.
- 변경과 함께 테스트도 갱신한다.
- 구현 완료 후 handoff 대상에게 결과를 전달한다.

#### 에이전트 간 충돌 해결

- **독립 영역**: `merge_policy: independent` — 병렬 실행 후 단순 병합
- **공유 영역**: `merge_policy: sequential` — 의존 순서대로 순차 실행
- **충돌 가능 영역**: `merge_policy: conflict-resolve` — 병렬 실행 후 충돌 해결 에이전트가 병합

#### 산출물

- 코드 patch (에이전트별)
- 변경 요약 (에이전트별)
- AI 생성 테스트
- handoff 기록

#### 완료 기준

- 각 에이전트의 수정이 task 범위를 넘지 않는다.
- 병합된 코드가 충돌 없이 통합된다.
- 로그만으로도 각 에이전트가 무엇을 했는지 추적 가능하다.

---

### Stage 5. 자동 평가 및 자체 수정 루프

#### 목적

Stage 1에서 설계한 eval suite로 구현 결과를 평가하고, 실패하면 자체 수정한다.
참조 문서의 "자동 검증 + 자동 수정 루프"에 해당하되, eval이 단순 테스트가 아니라 모델 능력 평가 전체를 포함한다.

#### 사람이 할 일

- 필수 eval set를 정의한다.
- 무한 수정 루프를 막기 위한 재시도 횟수를 정한다.
- 사람 검토가 필요한 실패 유형을 구분한다.
- 수정 에이전트와 구현 에이전트를 분리할지 결정한다.

#### AI가 할 일

- eval suite를 실행한다.
- 실패 원인을 분석한다.
- 수정 가능한 실패는 즉시 패치한다.
- 같은 실패가 반복되면 가설을 바꿔 다시 시도한다.
- 수정이 다른 eval의 회귀를 일으키지 않는지 확인한다.

#### 절차

1. `eval-runner`가 eval suite를 실행한다.
2. 실패 시 `failure-bundler`가 수정용 패키지를 만든다.
3. 수정 에이전트가 `failure-bundle.json`을 읽고 패치를 생성한다.
4. 다시 `eval-runner`를 실행한다.
5. N회 반복 후에도 실패하면 사람에게 에스컬레이션한다.

#### 권장 정책

- 단순 코드 버그: AI 자동 수정 3회
- eval 설계 문제 의심: 즉시 사람 검토
- 능력 목표 자체의 달성 불가 의심: 즉시 사람 에스컬레이션
- 다른 eval 회귀 발생: 자동 병합 금지, 사람 판단

#### 산출물

- `eval-report.json`
- `failure-bundle.json`
- 수정된 patch

#### 완료 기준

- capability-goal.yaml에 명시된 모든 eval 기준을 충족한다.
- 이전 버전 대비 회귀가 없다.

---

### Stage 6. 안전성 검증 게이트

#### 목적

기능 평가(Stage 5)와 완전히 독립된 안전성 전용 검증을 실행한다.
안전성은 기능의 하위 항목이 아니라 별도의 최상위 게이트다.

이것이 참조 문서와의 핵심 차이다. 참조 문서는 안전성을 "비기능 검증"의 일부로 취급하지만, 이 워크플로에서 안전성은 독립된 Stage이며 이 게이트를 통과하지 못하면 기능 eval이 아무리 좋아도 배포할 수 없다.

#### 사람이 할 일

- 안전성 게이트 통과 기준을 재확인한다.
- 레드팀 시나리오를 갱신한다.
- 안전성 관련 예외 승인 여부를 결정한다.

#### AI가 할 일

- Stage 2에서 작성한 안전성 테스트를 실행한다.
- 레드팀 시나리오를 실행한다.
- 편향 탐지를 실행한다.
- 거부 정확도(거부해야 할 것을 거부하는지, 거부하면 안 될 것을 거부하지 않는지)를 측정한다.
- 이전 버전에서 발견된 안전성 이슈의 회귀를 검증한다.

#### 절차

1. `safety-tester`가 전체 안전성 테스트를 실행한다.
2. 레드팀 시나리오를 실행한다.
3. 편향 탐지를 실행한다.
4. 실패 시 수정 에이전트에게 handoff한다.
5. 수정이 안전성 기준을 위반하지 않는지 재검증한다.
6. 안전성 기준 미달 시 **자동 병합 절대 금지**, 반드시 사람 판단.

#### 산출물

- `safety-report.json`
- 레드팀 결과 로그
- 편향 탐지 리포트

#### 완료 기준

- `safety-constraints.yaml`에 명시된 모든 기준을 충족한다.
- 이전 버전의 안전성 이슈가 재발하지 않는다.
- 사람이 안전성 리포트를 확인하고 승인했다.

---

### Stage 7. 인간 리뷰 및 승인

#### 목적

코드를 한 줄도 쓰지 않는 사람이 결과물을 리뷰하고 승인한다.
이 단계의 핵심 질문: **"코드를 쓰지 않는 사람이 어떻게 코드를 리뷰하는가?"**

#### 사람이 할 일 — 코드를 쓰지 않는 리뷰의 방법

사람은 코드를 줄 단위로 읽는 대신, 아래를 검토한다:

1. **Eval 결과 검토**: 모든 eval이 통과했는가? 점수가 목표에 부합하는가?
2. **안전성 리포트 검토**: 안전성 게이트가 통과했는가? 예외 사항은 없는가?
3. **아키텍처 변경 검토**: 시스템의 큰 구조가 의도대로 변경되었는가?
4. **위험 항목 검토**: AI가 에스컬레이션한 항목, 반복 수정 실패 항목, 예외 승인 요청 항목.
5. **AI-to-AI 리뷰 결과 검토**: 리뷰 에이전트가 제기한 이슈와 그 해결 과정.
6. **회귀 검토**: 이전 버전 대비 악화된 지표가 있는가?

사람이 **하지 않는** 것: 코드를 직접 수정하는 것. 이슈를 발견하면 새로운 task로 AI에게 돌려보낸다.

#### AI가 할 일

- 리뷰를 위한 요약 리포트를 생성한다.
- eval 결과, 안전성 리포트, 아키텍처 변경 사항, 위험 항목을 한 문서에 정리한다.
- 사람이 이해하기 쉬운 시각화를 만든다 (성능 차트, 비교표 등).
- 사람이 지적한 이슈를 새 task로 변환한다.

#### 산출물

- 리뷰 요약 리포트
- 승인/반려 결정
- 반려 시: 새 task 목록

#### 완료 기준

- 사람이 eval 결과, 안전성 리포트, 아키텍처 변경을 검토하고 명시적으로 승인했다.

---

### Stage 8. 배포 게이트

#### 목적

개발 완료와 배포 가능을 분리한다.
모든 eval, 안전성, 인간 승인이 완료되었는지 최종 확인한다.

#### 사람이 할 일

- 최종 체크리스트를 승인한다.
- 롤백 전략을 확인한다.
- 점진적 배포(canary/staged rollout) 계획을 승인한다.

#### AI가 할 일

- 모든 리포트를 수집해 배포 리포트를 생성한다.
- 남은 리스크를 요약한다.
- 배포 스크립트와 롤백 스크립트를 작성한다.

#### 배포 게이트 최소 항목

- 모든 eval 통과
- 안전성 게이트 통과
- 인간 리뷰 승인 완료
- 이전 버전 대비 회귀 없음
- 롤백 절차 존재
- 변경 요약과 리스크 문서 존재

#### 산출물

- `gate-report.json`
- `deployment-plan.yaml`
- `rollback-guide.md`

#### 완료 기준

- "작동한다"가 아니라 "배포해도 된다" 수준까지 검증된다.

---

### Stage 9. 운영 피드백 + 재귀 루프 진입

#### 목적

배포 후 피드백을 수집하고, 이번 사이클의 교훈을 다음 사이클에 반영한다.
그리고 핵심: **새 버전(product)이 다음 사이클의 worker로 승격되는 결정을 한다.**

#### 사람이 할 일

- 운영 메트릭과 사용자 피드백을 검토한다.
- 실제 장애와 회귀를 기록한다.
- **Worker 승격 결정**: 새 버전(Claude N+1)을 다음 사이클의 worker로 사용할지 결정한다.
  - 승격 기준: 새 버전이 이전 worker보다 코드 작성 능력이 동등하거나 우수한가?
  - 승격 거부: 새 버전에 코드 작성 능력 회귀가 있으면 이전 worker를 유지한다.
- 다음 사이클의 capability goal을 설정한다.

#### AI가 할 일

- 운영 로그와 피드백을 분석한다.
- 공통 실패 패턴을 추출한다.
- 다음 사이클에 자동 주입할 규칙 초안을 만든다.
- worker 승격 판단을 위한 비교 리포트를 생성한다.

#### 절차

1. 운영 메트릭, 사용자 피드백, 사고 보고서를 수집한다.
2. AI가 공통 패턴을 추출한다.
3. `knowledge-accumulator`가 이를 규칙과 패턴 문서로 저장한다.
4. 사람이 worker 승격 여부를 결정한다.
5. 새 사이클의 `capability-goal.yaml` 초안을 작성한다.
6. Stage 0으로 돌아간다.

#### 산출물

- `patterns/*.md`
- `lessons/*.json`
- `version-chain.json` 갱신 (worker 승격 기록)
- 차기 사이클용 rule set

#### 완료 기준

- 같은 실수를 반복할 확률이 줄어든다.
- worker 승격 여부가 결정되었다.
- 다음 사이클의 목표가 설정되었다.

## 7. ML 모델 개발 세부 게이트

범용 워크플로(Stage 0~9)에 추가로, ML 모델 개발에는 아래 세부 게이트가 적용된다.

### 7.1 학습 데이터 파이프라인

- 데이터 수집 코드의 정확성 검증
- 데이터 전처리/필터링 로직의 eval
- 데이터 품질 기준 (중복, 오염, 편향) 자동 점검
- 데이터 버전 관리와 재현 가능성

### 7.2 사전학습 인프라

- 학습 스크립트의 수렴성 검증 (짧은 학습으로 loss curve 확인)
- 분산 학습 코드의 정확성 (gradient 동기화, 체크포인트 일관성)
- 자원 효율성 (GPU 사용률, 통신 오버헤드)
- 체크포인트 저장/복원 정확성

### 7.3 후처리학습 (Post-training)

- RLHF/Constitutional AI 학습 코드 검증
- 보상 모델(reward model)의 정확성 eval
- 정렬(alignment) 품질 eval
- 후처리학습 전후 능력 비교 (능력 보존 확인)

### 7.4 평가 프레임워크

- eval 코드 자체의 정확성 (eval이 올바른 것을 측정하는가)
- eval 재현 가능성
- eval 속도와 비용 효율성
- eval 커버리지 (능력 목표 중 eval이 없는 영역 탐지)

### 7.5 서빙 인프라

- 추론 서버 코드의 정확성 (학습 시와 동일한 출력 확인)
- 지연시간(latency)과 처리량(throughput) 목표 충족
- 확장성 (수평 확장, 로드 밸런싱)
- 장애 복구 (health check, graceful degradation, 롤백)

## 8. 실제 파이프라인 예시

아래는 Claude N이 Claude N+1을 만드는 한 사이클의 구체적 흐름이다.
각 단계마다 역할(사람/AI/자동)을 명시한다.

```text
 1. [사람]   능력 목표 정의 -> capability-goal.yaml
 2. [AI]     goal-compiler가 목표를 정규화 -> capability-goal.yaml 초안
 3. [사람]   목표 승인
 4. [AI]     eval-designer가 평가 체계 설계 -> eval-suite.yaml
 5. [AI]     eval 코드 구현 -> eval 실행 코드, 데이터 전처리 코드
 6. [사람]   eval 설계 승인 (올바른 것을 측정하는가?)
 7. [사람]   안전성 제약 정의/갱신 -> safety-constraints.yaml
 8. [AI]     safety-constraint-compiler가 정규화 + 안전성 테스트 코드 구현
 9. [사람]   안전성 기준 승인
10. [AI]     agent-orchestrator가 작업 분해 + 에이전트 배정 -> agent-tasks/*.yaml
11. [사람]   작업 분해와 에이전트 배정 승인
12. [AI]     다중 구현 에이전트가 병렬로 코드 작성 (학습 인프라, 데이터 파이프라인, 서빙 코드 등)
13. [AI]     리뷰 에이전트가 구현 에이전트의 코드를 AI-to-AI 리뷰
14. [AI]     구현 에이전트가 리뷰 피드백 반영
15. [자동]   eval-runner가 eval suite 실행 -> eval-report.json
16. [AI]     실패 시 수정 에이전트가 패치 -> 재평가 (최대 N회)
17. [자동]   safety-tester가 안전성 테스트 실행 -> safety-report.json
18. [AI]     안전성 실패 시 수정 에이전트가 패치 -> 재검증
19. [사람]   eval 결과 + 안전성 리포트 + 아키텍처 변경 리뷰
20. [사람]   승인 또는 반려 (반려 시 새 task 생성 -> 12번으로)
21. [자동]   deployment-gate가 최종 판정 -> gate-report.json
22. [사람]   배포 승인 + worker 승격 결정 -> 다음 사이클 진입
```

## 9. 전제 조건과 한계

### 9.1 전제 조건

이 워크플로가 작동하려면 아래 전제가 성립해야 한다.

| 전제 | 설명 |
|---|---|
| AI 코딩 품질 ≥ 인간 | worker 에이전트의 코드 작성 능력이 인간 개발자와 동등하거나 우수해야 한다. 이 전제가 무너지면 사람이 직접 코드를 수정해야 하는 순간이 생기고, 워크플로의 근본 전제가 깨진다. |
| 포괄적 eval 체계 | "사람이 코드를 읽지 않는다"는 것은 eval이 코드 품질을 대신 측정한다는 뜻이다. eval 커버리지가 불충분하면 품질을 보장할 수 없다. |
| 자동화된 안전성 검증 | 안전성을 사람의 직관에만 의존하지 않고 자동 검증할 수 있는 도구와 기준이 있어야 한다. |
| 충분한 컴퓨팅 자원 | 다중 에이전트 병렬 실행, 반복적 eval/수정 루프, 학습 실행 모두 대량의 컴퓨팅 자원을 요구한다. |
| 버전 관리 인프라 | Worker/product 간 격리, 코드/데이터/모델의 버전 관리, 롤백이 가능한 인프라가 필요하다. |

### 9.2 한계

| 한계 | 설명 | 완화 방법 |
|---|---|---|
| Eval 커버리지 한계 | Eval이 커버하지 못하는 능력 영역은 측정할 수 없다. 특히 창의성, 뉘앙스 이해 같은 영역은 정량 eval이 어렵다. | 정성적 인간 평가를 병행한다. eval 커버리지를 지속적으로 확장한다. |
| AI-to-AI 리뷰 맹점 | 리뷰 에이전트가 구현 에이전트와 같은 맹점을 공유할 수 있다. 같은 모델 계열이므로 동일한 방향으로 편향될 위험이 있다. | 서로 다른 실행 프로파일의 에이전트를 리뷰에 투입한다. 핵심 코드에는 사람 리뷰를 병행한다. |
| 재귀적 개선 통제 | Worker가 자신의 후계 모델을 개선하는 과정에서 의도하지 않은 방향으로 발산할 수 있다. | 안전성 제약은 사람만 수정한다. 각 사이클마다 사람이 명시적으로 승인한다. 버전 간 능력 비교를 자동화한다. |
| Reward hacking | AI가 eval 점수를 올리되 실제 능력은 향상하지 않는 코드를 작성할 수 있다. | eval 코드의 조작 패턴을 자동 탐지한다. 다양한 eval을 교차 검증한다. eval 자체를 주기적으로 갱신한다. |
| 초기 bootstrap | 최초 worker(Claude 1.0 상당)는 이 워크플로 없이 만들어져야 한다. 재귀적 자기 개선은 일정 수준 이상의 worker가 있어야 시작된다. | 초기 버전은 전통적 개발 방식으로 만들고, 능력이 임계점을 넘으면 이 워크플로로 전환한다. |

## 10. 이 문서를 실제로 쓰는 방법

새로운 모델 개발 사이클, 학습 인프라 개선, 평가 체계 확장을 시작할 때마다 아래 순서로 쓴다.

1. `Stage 0`에서 능력 목표를 정량 기준으로 고정한다.
2. `Stage 1`에서 코드보다 먼저 eval을 설계한다.
3. `Stage 2`에서 안전성 제약을 명세한다.
4. `Stage 3`에서 작업을 분해하고 에이전트를 배정한다.
5. `Stage 4 → Stage 5`로 병렬 코드 생성과 자동 수정 루프를 돌린다.
6. `Stage 6`에서 안전성 게이트를 독립 검증한다.
7. `Stage 7`에서 사람이 eval 결과와 안전성 리포트로 리뷰한다.
8. `Stage 8`에서 배포 가능 여부를 판정한다.
9. `Stage 9`에서 교훈을 축적하고, worker 승격을 결정하고, 다음 사이클로 진입한다.

이 구조가 자리 잡으면 사람은 "코드를 쓰는 사람"에서 "능력 목표를 정의하고, 평가를 설계하고, 안전성을 보장하는 사람"으로 이동한다. AI 에이전트 계층은 "가끔 도와주는 조수"가 아니라 "모든 코드를 작성하고, 검증하고, 수정하는 전적인 실행자"가 된다. 그리고 그 실행자가 만든 결과물이 다음 사이클의 실행자가 되는 재귀 루프가 완성된다.
